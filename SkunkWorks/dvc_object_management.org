* Purpose

A sensible, end-to-end plan for managing the DVC-tracked objects in
LSMS_Library: raw microdata, documentation binaries, materialized
cross-country tables, local caches, and the stage pipelines that tie
them together.  The session that produced this document was a
planning session, not an execution session.  All the file edits and
commits described under [[#v070-execution-recipe][v0.7.0 execution recipe]] are deferred to a
follow-up session that can start fresh from this plan.

This plan absorbs and extends =SkunkWorks/dvc_writer_distribution.org=
(which is the executed design for write-access credential
provisioning) and cross-references =slurm_logs/v0.7.0_handoff.org=
(which is the already-scoped credential-path-and-wheel-bundling work).

* Scope and Principles

** DVC's role in this library

DVC has one job: *serve raw input data* (mostly =.dta=, some =.sav=
and =.csv=, plus the 2026-04-10 documentation migration) from a
central S3 bucket to any machine that holds a valid World Bank
Microdata API key.  Derived outputs --- harmonized cross-wave
parquets, panel-id JSON tables, cross-country aggregates --- are
*not* DVC's business.  They live in the user's =data_root()=
directory and are rebuilt on demand from inputs + library code.

The one exception to this historically was
=lsms_library/countries/Harmonized_LSMS-ISA_Ag/lsms_ag.parquet.dvc=,
a DVC-tracked derivative.  That exception is being dropped (see
[[#harmonized-lsms-isa-ag-drop][Harmonized_LSMS-ISA_Ag drop]] below) because the underlying project's
source is on GitHub and shouldn't be mirrored through our data
access layer.

** Guiding principles (restated and reaffirmed)

1. *DVC tracks heavy inputs, not derived outputs.*  If a file can
   be rebuilt from other files the library already knows about,
   don't DVC-track it.

2. *=data_root()= owns all user-visible derived artifacts.*  The
   repo tree is for source code, YAML configs, and a small set of
   .dvc pointer files.  Nothing the library materializes at runtime
   should land in the repo tree.

3. *Credential distribution is per-person IAM for writers, service
   account for readers.*  Readers get the S3 cache auto-unlocked on
   import via the WB API key + obfuscated GPG passphrase path.
   Writers go through =scripts/dvc_writer add USERNAME=, which
   provisions a per-person IAM user in the =dvc_writer= group.
   The real access gate is the WB Microdata Library ToU; the S3
   bucket is a cache over it, not an independent access layer.

4. *The repo tree stays small and grep-able.*  The 2026-04-10
   documentation migration (=10cd975c=) moved 35 large documentation
   binaries from git to DVC (~27 MB → 2.1 MB sdist, 4.9 MB wheel).
   Future heavy documents follow the same rule.

** 3. Materialized cross-wave / cross-country tables

Status: *inconsistent, mid-migration, subject of v0.7.0 and v0.8.0*.

This is the main subject of this plan.

*** What the library currently produces

For each =(country, table)= pair the library produces a Parquet
file like =household_roster.parquet= that stacks the country's
waves and applies harmonization.  The write path routes through
=_resolve_data_path= to =data_root(Country)/var/{table}.parquet=.

*** Three different code paths decide whether to *read* that cache

| Path                                              | Active when                                                                               | Invalidation                                                              |
|---------------------------------------------------+-------------------------------------------------------------------------------------------+---------------------------------------------------------------------------|
| =trust_cache=True=                                | Set explicitly on =Country(...)=                                                          | *None.*  Reads the file if present, no staleness check                    |
| =load_dataframe_with_dvc= (DVC stage layer)       | Country has a populated =dvc.yaml= (7 countries)                                          | DVC =stage.status()= hashes declared deps (.py files + country =_/= dir) |
| =load_from_waves= (fallback)                      | All other countries (~33)                                                                 | *None.*  Cache is *write-only*; every call rebuilds from source           |

The third row is the critical bug: for the majority of countries,
the =data_root()= parquet is written but never read back.  Every
call to =Country('Niger').household_roster()= re-runs
=load_from_waves()=, re-reads source =.dta= files, re-harmonizes,
rewrites the parquet, and returns.  Within a single Python process
a second call is faster because Country-instance ancillary caches
(=_sample_v_cache=, =_updated_ids_cache=, etc.) and the OS page
cache are warm, but across processes nothing persists.

Empirically verified 2026-04-10 on Savio n0065:
- Python import of =lsms_library= on Lustre: ~30s (dominates cold
  ipython UX)
- Niger =household_roster()= first call in a fresh process: ~22s
- Niger =household_roster()= second call, same process: ~8s (3x
  speedup from in-process state, not from parquet cache read)
- Two consecutive fresh processes: first call is ~22s *both
  times*; Lustre client cache doesn't persist across processes
- All reads go through the =local_file()= branch; zero =DVCFS=
  involvement (because =dvc pull= was run on this mirror)

Branch count on the second in-process call drops from 11 to 4,
confirming that ancillary lookups (panel_ids, sample, market
index) are cached but the four wave-level source files are
re-read and re-parsed every call.

*** The fresh-clone scenario is qualitatively different

The ~22s figure above is for a mirror with all =.dta= files already
on disk.  A fresh =pip install git+https://...@v0.7.0= user has no
=.dta= files locally and no =.dvc/cache/=.  Their flow:

1. =local_file()= → fails
2. =DVCFileSystem.open()= → streams from S3 (real network bandwidth)
3. Parse + harmonize + concat
4. Write to =data_root()= (write-only for the 33 countries)

First call: minutes (S3 bandwidth + parse).  Second call in a new
process: minutes again, because =data_root()= isn't being read
back.  This is the scenario v0.7.0 is supposed to enable, and
without a fix the user experience is "library works but is slow
on every single call."

*** Known incorrect documentation

=CLAUDE.md= previously claimed "Subsequent calls read cache (<1
sec). Caches auto-invalidate on source/config changes
(hash-based)."  Neither is true for the majority of countries as
of the v-migration.  =docs/guide/caching.md= had similar claims.
Both have been corrected in this session (see [[#doc-corrections][Doc corrections]]).

** 6. DVC stage layer (=dvc.yaml= materialize stages)

Status: *partial, inconsistent, retiring in v0.8.0*.

Countries with a populated =dvc.yaml=: Uganda, Senegal, Malawi (plus
5 wave-specific files under Malawi/), Togo, Kazakhstan, Serbia,
GhanaLSS.  7 total.  The root =lsms_library/countries/dvc.yaml= is
empty (={stages: {}}=).

For these 7 countries, =load_dataframe_with_dvc= in =country.py=
opens the country-level DVC repo, resolves the relevant stage,
calls =stage.status()= to check dirtiness, and either reads the
cached parquet or calls =stage.reproduce()=.  The =outs:= of these
stages point at in-tree =var/{table}.parquet=, which is why stale
in-tree parquets exist for Uganda and Senegal.

Problems:
1. Only 7 of ~40 countries.  Not a library-wide contract.
2. =outs:= in-tree contradicts the =data_root()= policy.
3. ~150 lines of =country.py= dedicated to the DVC path that
   mostly falls through to =load_from_waves= anyway.
4. Source =.dta= files are *not* in the stage deps, so editing
   raw data doesn't invalidate the stage.  Invalidation only
   triggers on edits to =cli.py=, =country.py=, =local_tools.py=,
   or the country's =_/= directory.

Plan:
- *v0.7.0*: leave the stage layer alone.  Removing it would cause
  a perf regression for the 7 countries that use it, because their
  current read-back only works through the stage layer.
- *v0.8.0*: once the in-house cache has proper hash-based
  invalidation, the stage layer becomes redundant.  Delete
  =load_dataframe_with_dvc=, delete =generate_dvc_stages.py=,
  delete the 7 =dvc.yaml= files, delete =run_stage.py=.
  Alternatively, keep =generate_dvc_stages.py= as a standalone
  reproducibility tool for generating =dvc.lock= files on demand
  (the "path iv" from the planning conversation) — TBD based on
  whether anyone is actually using it.

* v0.8.0 Roadmap

** Proper content-hash cache invalidation

Add a hash-based staleness check to =data_root()= cache reads.
Hash inputs should include:

- Content hash of source =.dta= / =.sav= / =.csv= files referenced
  from the wave's =data_info.yml= (via DVC pointer if present, via
  file content otherwise)
- Content hash of the wave's =data_info.yml=
- Content hash of any =_/{table}.py= scripts for that wave
- Maybe: content hash of =country.py=, =local_tools.py= (the
  library code that interprets the config) — or pin these by
  library version instead

Store the hash alongside the parquet, e.g. as
=data_root(Country)/var/{table}.parquet.hash=, check on read, fall
back to rebuild if mismatched.

** Retire the =dvc.yaml= materialize stage layer

Once hash-based invalidation is in place, the DVC stage layer
adds nothing that the in-house cache doesn't also provide, and
its inconsistency (7 vs 33 countries) becomes a net negative.

Delete:
- =load_dataframe_with_dvc= (country.py:1611-1761)
- The =_resolve_materialize_stages= method and =StageInfo= dataclass
- =_load_materialize_stage_map= and its =lru_cache=
- The 7 country-level =dvc.yaml= files
- =lsms_library/util/run_stage.py=
- Possibly =lsms_library/util/generate_dvc_stages.py= — or keep
  as opt-in reproducibility tool (see "path iv" decision below)
- The =LSMS_BUILD_BACKEND= env var handling, or keep as a no-op
  for backward compat for one release

Keep:
- =data_access.py= and the full raw-data fallback chain
- The =_/{table}.py= legacy script path (still needed for PP/PH
  countries and other irreducibly complex cases)

** Decision deferred: =generate_dvc_stages.py= disposition

Two positions from the planning conversation:

- *Delete*: it's legacy bookkeeping; nothing uses it at runtime
  after v0.8.0.
- *Keep as opt-in reproducibility tool*: contributors preparing a
  paper can generate =dvc.yaml= + =dvc.lock= on demand and commit
  it as a reproducibility artifact alongside the paper.  The
  runtime library ignores it.

Recommendation: keep-as-opt-in unless nobody wants it.  It's
~300 lines in =util/= and has no maintenance burden if not called
at runtime.

** Rethink =trust_cache=

After v0.7.0's simple "read if present" and v0.8.0's hash
invalidation, =trust_cache=True= becomes strictly weaker than the
default (reads without invalidation vs reads with invalidation).
Consider deprecating in v0.8.0 or v0.9.0 with a clear upgrade
path.

** Tooling gaps worth closing

- *CI check*: scan every =Country/Wave/Data/= directory for =.dta=
  files lacking a matching =.dvc= pointer (orphan source files
  are a common contributor mistake).  Fail the PR.
- *Periodic writer audit*: run =scripts/dvc_writer ls= monthly and
  prune stale members (anyone who hasn't touched the bucket in
  a year).  Could be a cron via =schedule= skill.
- *S3 bucket size monitoring*: no current visibility into
  =dvcbucket0= size and growth rate.  Add a weekly report.
- *=_log_issue= pollution (#148)*: =country.py:168= appends to
  =lsms_library/ISSUES.md= on every build failure, polluting the
  working tree.  Either redirect to =slurm_logs/= or gitignore
  the file.
