#+title: Contributing

* Contributing New Data
** Prerequisites

  Things you'll want to have set up in advance include:

  - A =github.com= account.  You'll be much happier if you upload
    your public =ssh-key= to =github.com= for authentication
    purposes.  This is the service that will track changes to source
    code or other documents, other than those which should be
    ephemeral (e.g., log files) or files managed by =dvc= (see below).

  - The =git= binary on your local computer.

  - =dvc=.  See https://dvc.org/doc.  This is used to manage static,
    large files (which aren't a good fit for =git=).  You'll also need
    to install dependencies which allow =dvc= to work with google
    drive and s3 buckets.  Provided you have a reasonably modern =python=
    installation (if you don't we recommend
    https://www.anaconda.com/), a simple  =pip install 'dvc[s3]'= may do the trick.  Some shells (e.g., the default zsh on Macs) may require you to escape the square brackets, e.g., =pip install dvc\[s3\]=.

  - =gnupg=, for managing encryption.

  - =emacs=, for working with the =org-mode= files used in this project.


** Setting Up the LSMS Library
  Follow the following steps to use or add to the LSMS Library.

*** Fork or Clone
   Optionally fork (creating your own version), then clone (using
   either your own or =eligon='s version) the repo to your local
   machine, using 
   #+begin_src sh
   git clone git@github.com:ligon/LSMS_Library.git LSMS_Library
   #+end_src
   (Note the capitalization of LSMS_Library, which we'll assume for
   the purposes of this document).  If you fork, then you should
   replace the "eligon" string in the above with your own username.

   This step will set up all the files managed by git in the local
   directory =LSMS_Library= on your machine.

*** DVC Setup
   DVC manages the versioning of more-or-less static datasets.
   To install,
   #+begin_src sh
   pip install dvc[s3]
   #+end_src
The private key for accessing the data on google drive is found in the file =gdrive_data_store.json.gpg=.  The =gpg= string indicates that this file is encrypted, in this case using a secret passphrase.

Similarly, the private key for /accessing/ data on the s3 data store is found in =s3_reader_creds.gpg=.  This is encrypted using the same passphrase mentioned above. To learn the secret passphrase you'll need to ask
   =ligon@berkeley.edu=.  Once you have it:    


   *Authentication via Python*   
   
   Use the helper function provided in the LSMS Library:   
   
   #+begin_src python
   import lsms_library as ll
   ll.authenticate()
   #+end_src
   
   You’ll be prompted to enter the secret passphrase (which you can request from =ligon@berkeley.edu=).  
   If successful, this will unlock the credentials needed for DVC and allow you to access the data in the LSMS Library.

   (If this doesn't work, make sure you have =gpg= installed: https://gnupg.org/download/)

   *User configuration*

   Library settings (including the World Bank Microdata API key) live in
   =~/.config/lsms_library/config.yml=:

   #+begin_src yaml
   microdata_api_key: your_key_here
   # data_dir: /path/to/override   # same as LSMS_DATA_DIR env var
   #+end_src

   For each setting, an environment variable (e.g., =MICRODATA_API_KEY=,
   =LSMS_DATA_DIR=) takes precedence over the config file.  The
   =ll.authenticate()= helper sets up DVC credentials; the config file
   handles everything else.

   In non-interactive environments (e.g., CI) you can suppress the automatic authentication prompt by setting
   =LSMS_SKIP_AUTH=1= in the environment. When this flag is set you are responsible for running =ll.authenticate()=
   manually before attempting to access protected datasets.

*** Local virtual environment
   Keep project dependencies isolated so you can experiment without colliding with
   system packages:

   #+begin_src sh
   cd LSMS_Library
   python3 -m venv .venv
   source .venv/bin/activate
   pip install --upgrade pip
   pip install -e .
   pip install pytest ipython matplotlib   # developer conveniences
   #+end_src

   When running project tools (e.g., =make= targets), either activate the
   environment first or prepend the virtualenv to your PATH, e.g.

   #+begin_src sh
   PATH="$(pwd)/.venv/bin:$PATH" make -C lsms_library test country=Malawi
   #+end_src

   The editable install exposes =lsms_library= so the import hooks and
   authentication helpers work consistently inside the virtualenv.
* Streaming dvc files
   A =dvc pull= will download dvc files to your local repository.
   But this is usually not the best way to proceed!  In particular, =dvc=
   offers an api which permits one to "stream" or cache files, leaving
   your storage local to the working repository free of big data
   files.

   To illustrate,
   #+begin_src python
     import dvc.api
     import pandas as pd

     with dvc.api.open('BigRemoteFile.dta',mode='rb') as dta:
         df = pd.read_stata(dta)
   #+end_src
   This will result in a =pandas.DataFrame= in RAM, but will use no
   additional disk (except that, depending on what's being used as the
   dvc store, the file may actually be stored in =.dvc/cache=; this
   cache can be cleared with =dvc gc=).

** Pulling dvc files
   If you need the actual file instead of a "stream" you can instead
   "pull" the dvc files, using
   #+begin_src sh
   dvc pull
   #+end_src
   and files should be added from the remote dvc data store to your
   working repository. 

** Persisting harmonized tables
   When you need reproducible Parquet/CSV artefacts, materialise them through the
   public API instead of re-implementing transformations. There are three entry
   points:

   - CLI (materialize): =python -m lsms_library.cli materialize --country Malawi --wave 2019-20 --table household_roster --format parquet --out lsms_library/countries/build/Malawi/2019-20/household_roster.parquet= (or use the installed script: =lsms-library materialize ...=). Repeat =--wave= to pull several waves, or omit it / pass =--all-waves= to stack every available wave into a single file (e.g., =lsms-library materialize --country Malawi --table household_roster --out lsms_library/countries/build/Malawi/all_waves/household_roster.parquet=).
   - CLI (register): =python -m lsms_library.cli register --country Uganda --wave 2023-24 --table household_roster= appends the appropriate entry to =lsms_library/countries/dvc.yaml= and clears any stale lock data.
   - CLI (discovery): =lsms-library countries= lists supported countries; =lsms-library waves --country Malawi= shows available waves; =lsms-library tables --country Malawi= lists tables from the data scheme.
   - Makefile: =make materialize country=Malawi wave=2019-20 table=household_roster= writes to =lsms_library/countries/build/<country>/<wave>/<table>.parquet=. Provide multiple waves via =wave=\"2019-20 2021-22\"= or aggregate everything with =all_waves=true=; override =OUT_FILE= to customise the destination. =make register ...= invokes the helper above (still requires a specific wave to keep DVC stages reproducible).
   - DVC: reproducible stages live in =lsms_library/countries/dvc.yaml=. Run =cd lsms_library/countries= first, then =dvc repro materialize@malawi_2019_20_household_roster=. The Makefile target =make persist ...= wraps the same command from the project root.

   When you add new tables, extend (or simply rerun =lsms-library register ...= to update) =lsms_library/countries/dvc.yaml= so contributors can regenerate all cached outputs with a single =dvc repro=.

   Both the CLI and the DVC stages stream source assets via DVC's API, so you do not need to "dvc pull" large datasets locally—ideal for light machines.
   Tip: run =lsms-library --install-completion= to enable shell tab-completion (Typer/Click feature).

   Use =lsms-library waves --all-countries= or =lsms-library tables --all-countries= for CSV-friendly country combinations.

   After editing metadata or adding YAML/Python loaders for a country, keep the DVC files in sync and ensure caches stay coherent:

   - Regenerate the =dvc.yaml= definitions from the canonical metadata with:
     #+begin_src bash
poetry run python -m lsms_library.util.generate_dvc_stages --countries Uganda
     #+end_src
   - Refresh the matching lockfile (and therefore the validated cache) by reproing the specific stage:
     #+begin_src bash
cd lsms_library/countries
poetry run dvc repro Uganda/2019-20/dvc.yaml:materialize@uganda::2019_20::household_roster
     #+end_src
   - If you only need to sanity-check a table without touching DVC metadata, set =LSMS_BUILD_BACKEND=make= and call either =lsms-library materialize ...= or the per-country Makefile target; the environment variable keeps the runtime loader from re-entering DVC.
   - Use =lsms-library cache clear --country <Country>= (plus =--method= / =--wave= filters) before rerunning DVC stages when you want to guarantee that fresh parquet/JSON artefacts are produced.


* Adding New Data
** Capturing Household Location (undergrad workflow)
   Several downstream analyses need household-level location metadata (rural/urban,
   region, cluster id, coordinates). Undergrads can add this without touching any
   Python code by editing only the YAML metadata and letting our existing generator
   do the rest.

   - Schema update ::
     edit =lsms_library/countries/<Country>/_/data_scheme.yml= and add:
     #+begin_src yaml
     location:
       index: (i, t)
       cluster_id: str
       Region: str
       Rural: str
       latitude: float
       longitude: float
     #+end_src
     Adjust column names/types to match what that country actually has.

   - Wave metadata ::
     for each wave (=lsms_library/countries/<Country>/<wave>/_/data_info.yml=) add
     a stanza describing the source columns, for example:
     #+begin_src yaml
     location:
       source: household_characteristics.parquet
       cluster_column: EA_ID
       region_column: Region
       rural_column: Rural
     #+end_src
     If a wave lacks some of these fields, leave the entry blank so the gap stays visible.

   - Regenerate DVC ::
     keep `dvc.yaml`/`dvc.lock` aligned with:
     #+begin_src sh
     poetry run python -m lsms_library.cli generate-dvc --country <Country>
     #+end_src
     Commit those updates along with your YAML changes.

   - Optional sanity check ::
     from =lsms_library/countries/= reproduce the DVC stage:
     #+begin_src sh
     cd lsms_library/countries
     poetry run dvc repro <Country>/dvc.yaml:materialize@<country_slug>::location
     #+end_src
     This confirms the lock file and cache update cleanly.

   These four steps keep the work scoped entirely to `data_scheme.yml` and
   `data_info.yml`, which means contributors don’t need to touch the Python codebase.
   If you notice that a wave lacks the required columns, record that fact in the
   YAML (leave the fields blank) rather than inventing data—the generator will emit
   empty columns until real mappings are added.

** Procedure to Add Data
   To add a new LSMS-style survey to the repo, you'll follow the
   following steps.  Here we give the example of adding a 2015--16
   survey from Uganda, obtained from
   https://microdata.worldbank.org/index.php/catalog/3460.  The same
   steps should work for you /mutatis mutandis/:

  1. Create a directory corresponding to the country or area; e.g., 
     #+begin_src sh
     mkdir Uganda
     #+end_src
  2. Create a /sub/-directory indicating the time period for the
     survey; e.g., 
     #+begin_src sh
     mkdir Uganda/2015-16
     #+end_src
  3. Create a =Documentation= sub-directory for each survey; e.g.,
     #+begin_src sh
     mkdir Uganda/2015-16/Documentation
     #+end_src
     In this directory include the following files:
     - SOURCE :: A text file giving both a url (if available) and
       citation information for the dataset.
     - LICENSE :: A text file containing a description of the license
       or other terms under which you've obtained the data.
  4. Add other documentation useful for understanding the data to the
     =Documentation= sub-directory.

  5. Add all the contents of the =Documentation= folder to the =git= repo;
     e.g., 
     #+begin_src sh
     cd ./Uganda/2015-16/Documentation
     git add .
     git commit -m"Add Uganda 2015-16 documentation to repo."
     git push
     #+end_src

  6. Create a =Data= sub-directory for each survey; e.g.,
     #+begin_src sh
     mkdir Uganda/2015-16/Data
     #+end_src

  7. Obtain a copy of the data you're interested in, perhaps as a zip
     file or other archive.  Store this in some temporary place, and
     unzip (or whatever) the files into the relevant Country/Year/Data
     directory, taking care to preserve any useful directory structure
     in the archive.  E.g.,
     #+begin_src sh
     cd Uganda/2015-16 && unzip -j /tmp/UGA_2015_UNPS_v01_M_STATA8.zip
     #+end_src
  8. Add the data you've unarchived to =dvc=, then add the /pointers/
     (i.e., files with a .dvc extension to git).  For the Uganda case we assume that
     all the relevant data comes in the form of =stata= *.dta files,
     since this is what we downloaded from the World Bank.  For example,
     #+begin_src sh
     cd ../Data
     dvc add *.dta
     git commit -m"Add Uganda/2015-16/Data/*.dta files to dvc store."
     git pull && git push
     #+end_src
  9. Push the data files to the dvc store. Make sure you have good
     internet connection!  Then a simple
     #+begin_src sh
     dvc push
     #+end_src
     will copy the data to the remote data store.  NB: If this is the
     first time you've done this for this repository, then you'll
     first need to jump through some simple hoops to authenticate with
     gdrive.
  10. With the files pushed to the dvc store, you won't need them
      locally anymore, so you can do something like
      #+begin_src sh
      cd ../Data && rm *.dta
      #+end_src
      or (if you have a more complex directory structure) perhaps
      #+begin_src sh
      find ../Data -name \*.dta -exec rm \{\} \;
      #+end_src
* Adding New Data Artifacts Using YAML

This guide explains how to organize ~data_scheme.yml~, ~data_info.yml~, and optional Python helpers so the loader can build tidy DataFrames for each survey wave. Keep indentation consistent—YAML is picky, and a slipped space will raise a parsing error.

** Directory Layout

Every country lives under ~lsms_library/countries/<Country>~. Inside each country you’ll find a shared ~=_/= directory for cross-wave settings and one subdirectory per wave:

#+begin_example
Malawi/
  _/
    data_scheme.yml
    malawi.py        # optional country-wide helpers
  2019-20/
    _/
      data_info.yml
      2019-20.py     # optional wave-specific helpers
    Data/
      … Stata/Parquet files tracked by DVC …
#+end_example

**  Describe Tables in ~data_scheme.yml~

The country-level scheme names each standardized table, lists its multi-index, and defines column types. Use the same keys the Python API exposes (e.g., ~food_acquired~, ~household_roster~). A concise example from ~lsms_library/countries/Malawi/_/data_scheme.yml:1~:

#+begin_src yaml
Country: Malawi

Data Scheme:
  household_roster:
    index: (t, i, pid)
    Sex: str
    Age: int
    Generation: int
    Distance: int
    Affinity: str
  food_acquired:
    index: (t, v, visit, i, j, u)
    Quantity: float
    Price: float
    Produced: float
#+end_src

Keep index labels aligned with the loaders in ~lsms_library/local_tools.py~; ~t~ is the survey wave, ~i~ household id, ~v~ cluster, and so on.

Required columns and accepted values are defined in the canonical schema ~lsms_library/data_info.yml~. The test suite validates all ~data_scheme.yml~ files against it.

**  Author ~data_info.yml~ for Each Wave

Each wave’s ~=_/data_info.yml~ maps survey files to the scheme. Begin with ~Country~ and ~Wave~, then add one top-level key per table listed in the scheme. For each table define:

- ~file~: path(s) to source data relative to the wave directory. Provide a single string or a list. Use dictionaries inside the list to override column mappings for a specific file.
- ~idxvars~: how to construct each index column.
- ~myvars~: value columns.

***  Basic Structure

From ~lsms_library/countries/Mali/2021-22/_/data_info.yml:4~:

#+begin_src yaml
household_roster:
  file: ehcvm_individu_mli2021.dta
  idxvars:
    v: grappe
    i:
      - grappe
      - menage
    pid:
      - grappe
      - menage
      - numind
  myvars:
    Sex: sexe
    Age: age
    Relationship: lien
#+end_src

Lists inside ~idxvars~ signal a composite key. The loader passes that list into ~local_tools.df_data_grabber~ to build the final identifier.

Note that ~Relationship~ in ~myvars~ produces a string column (e.g., "Head", "Spouse", "Child"). The runtime automatically decomposes this into ~Generation~, ~Distance~, and ~Affinity~ columns (see /Kinship Decomposition/ below).

*** Value Mappings and Inline Fixes

You can normalise categorical labels or repair known errors inline. The Uganda 2005–06 roster (~lsms_library/countries/Uganda/2005-06/_/data_info.yml:32~) shows both comment style and a mapping dictionary:

#+begin_src yaml
cluster_features:
  file: GSEC1.dta
  idxvars:
    i: HHID
    v: comm
  myvars:
    Region:
      - region
      -
        0: "Central"   # Kampala households coded as 0, remap to "Central"
    Rural:
      - urban
      -
        urban: Urban
        0: Rural
#+end_src

Leave a blank line between the column name and its dictionary so YAML treats the mapping as part of the list.

*** Referencing Categorical Mapping Tables

For large synonym lists, link to a pre-built mapping table instead of hard-coding values. The Mali food acquisition block (~lsms_library/countries/Mali/2021-22/_/data_info.yml:29~) uses the ~mappings~ helper to pull replacements from ~categorical_mapping/harmonize_food~:

#+begin_src yaml
food_acquired:
  idxvars:
    j:
      - s07bq01
      - mappings: ['harmonize_food', 'Original Label', 'Preferred Label']
    u:
      - s07bq03b
      - mappings: ['unit', 'Original Label', 'Preferred Label']
#+end_src

The loader fetches the named table, then aligns the “Original Label” column to the canonical “Preferred Label”.

*** Multiple Source Files and Overrides

Use a list under ~file~ to stitch together cross-sectional and panel inputs. You can override column mappings for a specific file by supplying a dictionary. Malawi 2019–20 (~lsms_library/countries/Malawi/2019-20/_/data_info.yml:4~) illustrates both patterns:

#+begin_src yaml
household_roster:
  file:
    - Cross_Sectional/HH_MOD_B.dta
    - Panel/hh_mod_b_19.dta:
        i: y4_hhid
        pid: id_code
  idxvars:
    i:
      - case_id
      - function: cs_i   # apply only when case_id is used
    pid: PID
#+end_src

*** Boolean Flags and Other Transformations

Add a ~mapping: {<raw>: <clean>}~ block to coerce values, or point to a function when the built-ins are insufficient. In the previous example, ~function: cs_i~ calls a helper defined in ~lsms_library/countries/Malawi/_/malawi.py~. If every source file needs the same formatter, name the helper after the column (e.g., define ~def i(...):~) and skip the explicit ~function~ key.

** Kinship Decomposition

The ~household_roster~ table uses a principled kinship decomposition (Kroeber 1909; Romney & D'Andrade 1964) instead of a single ~Relationship~ string. Four columns capture household relationships without redundancy:

| Column       | Type | Description                                                          |
|--------------+------+----------------------------------------------------------------------|
| ~Sex~        | str  | ~M~ or ~F~                                                           |
| ~Generation~ | int  | Vertical distance from head (0=same, +1=parent, -1=child, etc.)      |
| ~Distance~   | int  | Collateral distance (0=lineal, 1=sibling line, 2=cousin line)        |
| ~Affinity~   | str  | ~consanguineal~, ~affinal~, ~step~, ~foster~, ~unrelated~, ~guest~, ~servant~ |

This means:
- ~Sex~ is never redundant with the relationship label ("Brother" = Sibling + M).
- Polygyny is handled naturally: all wives are ~(0, 0, affinal)~.
- Foster children ~(-1, 0, foster)~ are distinguished from step children ~(-1, 0, step)~ and adopted children ~(-1, 0, consanguineal)~.

*** How it works

Per-wave ~data_info.yml~ files produce a ~Relationship~ string from source data (e.g., ~Relationship: lien~). The runtime automatically decomposes it into the three columns using a dictionary in ~lsms_library/categorical_mapping/kinship.yml~:

#+begin_src yaml
Kinship:
  Head:                [0, 0, consanguineal]
  Spouse:              [0, 0, affinal]
  Child:               [-1, 0, consanguineal]
  Parent:              [1, 0, consanguineal]
  Sibling:             [0, 1, consanguineal]
  Cousin:              [0, 2, consanguineal]
  Parent-In-Law:       [1, 0, affinal]
  Foster Child:        [-1, 0, foster]
  Servant:             [0, 0, servant]
  # ... 58 entries total
#+end_src

*** Adding new relationship labels

If a survey uses a relationship label not yet in the dictionary, the library emits a warning:

#+begin_example
UserWarning: Unknown relationship labels (add to spelling.yml Kinship): ['Beau-fils']
#+end_example

To fix: add the label to ~kinship.yml~ with its ~[Generation, Distance, Affinity]~ tuple:

#+begin_src yaml
  Beau-fils:           [-1, 0, affinal]
#+end_src

*** Usage examples

#+begin_src python
import lsms_library as ll
mali = ll.Country('Mali')
roster = mali.household_roster()

# All children in the household
children = roster[roster.Generation == -1]

# All spouse(s), including polygynous households
spouses = roster[(roster.Generation == 0) & (roster.Affinity == 'affinal')]

# Extended family (collateral distance > 0)
extended = roster[roster.Distance > 0]

# Non-kin members
non_kin = roster[roster.Affinity.isin(['unrelated', 'guest', 'servant'])]
#+end_src

** Canonical Schema and Spelling Enforcement

The file ~lsms_library/data_info.yml~ defines the canonical schema for all cross-country tables:

- *Required columns* per table (e.g., ~household_roster~ must have ~Sex~, ~Age~, ~Generation~, ~Distance~, ~Affinity~).
- *Accepted values* for constrained columns (e.g., ~Sex: [M, F]~).
- *Rejected spellings* for column names (e.g., ~Relation~ → use ~Generation, Distance, Affinity~; ~Effected~ → ~Affected~; ~Highest_education~ → ~Educational Attainment~).

The test suite (~tests/test_schema_consistency.py~) validates all country ~data_scheme.yml~ files against this canonical schema. When adding new tables or columns, update ~data_info.yml~ first, then the per-country files.

Cross-country value normalisation is declared directly in ~data_info.yml~ using ~spellings~ inverse dictionaries on column definitions. Each key is the canonical value; its list contains accepted variant spellings that the runtime maps to the canonical form:

#+begin_src yaml
Sex:
  type: str
  required: true
  spellings:
    M: [Male, male, Masculin, masculin, Homme, homme, m]
    F: [Female, female, Féminin, feminin, Femme, femme, f]
#+end_src

The canonical values are simply the dictionary keys. The runtime replaces variants with canonical forms automatically when data is loaded.

The kinship decomposition dictionary lives in ~lsms_library/categorical_mapping/kinship.yml~. Within-country value harmonisation uses ~categorical_mapping.org~ files per country.

** Python Helpers for Advanced Cases

Some datasets need more than simple mappings—row-level fixes, custom joins, or Makefile-driven builds. You can:

- Create ~<Country>/_/<country>.py~ for helpers shared across waves.
- Create ~<Country>/<Wave>/_/<Wave>.py~ for wave-specific logic.
- Provide a function named after the table (e.g., ~def food_acquired(df):~) to post-process the assembled DataFrame.
- For very custom builds, drop (or add to) a ~Makefile~ into the wave or country directory; tables tagged ~# USE_MAKEFILE~ in ~data_scheme.yml~ will invoke it.

** Validate Your YAML

Before opening a pull request:

1. Confirm the YAML parses:
   #+begin_src sh
   python -c "import yaml, pathlib; yaml.safe_load(pathlib.Path('lsms_library/countries/Malawi/2019-20/_/data_info.yml').read_text())"
   #+end_src
2. Load the table via the library (replace ~Malawi~ and ~household_roster~ with your targets):
   #+begin_src python
   import lsms_library as ll
   mw = ll.Country('Malawi')
   df = mw.waves['2019-20'].household_roster()
   print(df.head())
   #+end_src
3. Run ~make test country=<Country>~ from ~lsms_library~ to exercise the harmonizer.

Capturing the exact commands you ran (and any gotchas) in your pull request keeps future maintainers out of the tall grass.

Note: the ~make test~ target expects the relevant Parquet/Stata assets to be
available through DVC. Ensure you've authenticated and pulled/streamed the
required files before relying on the smoke test.
