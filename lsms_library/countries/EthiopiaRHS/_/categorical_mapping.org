#+title: EthiopiaRHS categorical mappings

Cross-wave harmonization tables.  ERHS label *spellings* drift across
files/rounds (e.g. village code 6 is "Shumsha" in demo123 but
"SHUMSHEHA" in food1) while the numeric *codes* are stable.  So every
ERHS wave extracts with =converted_categoricals: False= and maps the
stable codes here.  Referenced from each wave's =data_info.yml= via
the =mappings:= list form, e.g.

: Relationship:
:     - q11_1b
:     - mappings: ['erhs_relation', 'Code', 'Preferred Label']

* erhs_relation

Relationship-to-head codes (questionnaire q11_1b / round-2/3
equivalents share this code scheme).  Preferred Label strings match
the kinship.yml keys added for GH #271 so =_expand_kinship= decomposes
them into (Generation, Distance, Affinity).

Code -9 ("Head Died and Replaced", round-4 reltn4 interview-status
sentinel) and any other non-1..16 value are not real relationships;
they map to an empty Preferred Label so =_expand_kinship= treats them
as genuine-missing (no spurious kinship warning), same as NaN.

* erhs_relation_1989

The 1989 pre-expansion instrument (demog89_1.dta =relnhhed=) uses a
DISTINCT 9-code relationship taxonomy, NOT the 16-code =erhs_relation=
of the 1994+ rounds.  Mapped to Preferred Labels that already resolve
in kinship.yml.  Judgment calls: code 5 "Illegitimate child" -> child
(consanguineal, like Own child); code 8 "Helper" -> Servants; code 9
"Other" -> Other unrelated person (conservative; could be distant kin
or non-kin -- treated as unrelated rather than guessing a kin tie).

* harmonize_food

+ Aggregate + one CODE column per wave).  ERHS has no stable food
code scheme -- 1994b/1995/1997 share one numbering (1997 labels
truncated), 1989 and 1994a each have their own -- so each wave's
data_info maps its own code column via this table:
  j: [<codecol>, mappings: ['harmonize_food', '<wave>', 'Preferred Label']]
First-pass reconciliation (normalized-label + shared-code
alignment); refinements (variant splits/merges) are a documented
follow-up, mirroring how Uganda's table evolved.  Use
labels='Aggregate' for the coarse grouping.

* harmonize_unit

Unit code -> canonical label.  The ERHS unit code scheme is
STABLE across the 1994+ waves (only spelling drifts), so a single
Code column suffices (cf. harmonize_food which needed per-wave
columns).  Metric & good-invariant codes map to framework-
recognized tokens (1->kg, 2->"100 kg" so the explicit-metric
parser yields 100, 19->litre, 92->g) and 30->Birr (currency:
shared, intentionally no kg factor).  ALL OTHER units are local
volumetric/containers whose kg-equivalent is good-specific; the
food_acquired df_edit appends the food label (e.g. "Chinet/Butter")
so the framework price-ratio inference recovers a per-good factor
(metric units stay shared -- a kg is good-invariant).  1989
unitcode is a different, in-data-unlabelled scheme -> 1989 units
stay raw+good-specific (no metric distinction) pending Codeunit1.SPS.
