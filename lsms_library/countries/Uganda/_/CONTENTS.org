* Survey Program

The Uganda National Panel Survey (UNPS) is conducted by the Uganda
Bureau of Statistics (UBOS) with support from the World Bank
Living Standards Measurement Study (LSMS).  It builds on the
Uganda National Household Survey (UNHS) 2005/06 baseline.

- World Bank catalog :: https://microdata.worldbank.org/index.php/catalog/1001

* Sampling Design
** Overview

Two-stage stratified cluster sample.  Enumeration Areas (EAs) are
the primary sampling unit (PSU), drawn from the 783 EAs visited in
the UNHS 2005/06.  The UNPS selected 322 EAs covering 3,123
households.

The UNPS uses two visits per year (six months apart) to capture
both agricultural seasons.

** Cluster (PSU) identifier

The cluster identifier (=v= in the library's index convention) also
changed:

| Waves       | Variable      | Type    | Description            |
|-------------+---------------+---------+------------------------|
| 2005--2012  | =comm=        | numeric | Community/EA code      |
| 2013-14     | =h1aq4a=      | numeric | Parish code (8 digits) |
| 2015-16     | =parish_code= | numeric | Parish code            |
| 2018-19     | =parish_name= | string  | Parish name            |
| 2019-20     | =s1aq04a=     | string  | Parish name            |

All are drawn from GSEC1 (the household cover page).  The shift
from numeric EA codes to parish names occurred around 2015-16 as
the sampling frame evolved from strict EA boundaries to parish-level
clusters to accommodate split-off households.

*** Hybrid =v= in 2009-10 (movers and split-offs)

The 2009-10 =comm= column is labeled /Current EA Code as in 2005/06
Data/: it is the /original 2005-06/ sampling-frame EA, not a new
2009-10 code.  That convention makes sense for panel continuity
(every /original/ 2005-06 household keeps its EA across waves) but
leaves two groups of 2009-10 households without a usable =comm=:

| =hh_status= | count | why =comm= is blank                                     |
|-------------+-------+---------------------------------------------------------|
| 2 (mover)   |   192 | Left the original EA; no 2005-06 code applies.          |
| 3 (split-off) | 368 | Did not exist in 2005-06; no 2005-06 code applies.     |
| 1 (orig. non-mover, anomaly) | 5 | Isolated data-entry gap (HHIDs =3163001601..1608=). |
| Total       |   565 | 19% of 2 975 interviewed households.                    |

These 565 households are not a data bug — by survey design they are
/outside the 2005-06 sampling frame/ that =comm= encodes.  They do,
however, carry modified GPS coordinates from =UNPS_Geovars_0910.dta=
(=lat_mod=/=lon_mod=, jittered ±2 km urban / ±5 km rural, with a
=source= flag distinguishing current-location from origin-location
fixes), so a within-wave spatial identifier can still be built.

To avoid silent data loss in downstream tables (the old behaviour
dropped these 565 households in =roster_to_characteristics='s
=groupby=), =sample()='s =v= column in 2009-10 is populated by a
hybrid rule:

- =v = comm= (7-digit numeric string) when =comm= is non-blank —
  2 410 households, 317 distinct panel EAs.  Matches CSECTION1's
  =comm= index exactly, so community-questionnaire joins work as
  they always did.
- =v = "@{lat:+.2f},{lon:+.2f}"= (a /synthetic/ label prefixed with
  =@=, binned to a 0.05° grid ≈ 5.5 km) when =comm= is blank but
  coordinates are available — 541 households, ~339 distinct bins.
  These labels do /not/ match any CSECTION1 =comm=, by design: the
  community questionnaire was not administered at the movers' or
  split-offs' 2009-10 locations, so a non-match is the correct
  outcome of a join.
- =v = <NA>= when neither =comm= nor coordinates are available —
  24 households, the refreshment / geo-missing tail that is also
  reported in the =UNPS_Geovars_0910= docs as /Invalid: 24/.

Implementation: in =data_info.yml= the =sample:= block uses the
=dfs:=/=derived:=/=drop:= pattern to merge GSEC1 and the geovars
file, then dispatches to the =coalesce_coord_bin= transformer in
=lsms_library/transformations.py=.  The 0.05° grid resolution and
=@= prefix are documented defaults on the transformer and can be
overridden in the YAML if another wave or country needs different
values.

Why only 2009-10 gets the hybrid treatment: 2010-11 and 2011-12
also use =comm= from GSEC1, but their raw surveys populated it for
every household (movers and split-offs got re-coded with their
original-sample EA), so no fallback is needed.  From 2013-14 onward,
=v= is a parish identifier — every household has one — and the
issue disappears by construction.

*** HH-level =Region= on =sample()=

Since =v= can be a synthetic =@lat,lon= label (§above) that has no
entry in =cluster_features()=, consumers that needed =Region= used to
pull it from =cluster_features()= and silently lose those households
— the cluster-level table is keyed on real EA codes and cannot answer
for synthetic ones.  =sample()= now exposes =Region= directly as a
household-level column for every Uganda wave (it reads the same
=region= field that =cluster_features()= does, with the same Kampala
→ Central fold in 2009-10 and 2010-11 where applicable).

Practical consequence: =Country('Uganda').sample()[['v','Region']]=
is complete for every household; joining =Region= from =sample()=
— rather than =cluster_features()= — avoids dropping the 565
movers/split-offs from any =(i, t, m)=-keyed regression where =m=
is derived from Region.  Recommended pattern:

#+begin_src python
uga = ll.Country('Uganda')
d = uga.household_characteristics()
# Region = market at HH level, available for all HH including movers
d = d.join(uga.sample()[['Region']])
d = d.rename(columns={'Region': 'm'}).set_index('m', append=True)
#+end_src

Continue to use =cluster_features()[['Region']]= when you want the
cluster-level view (e.g., summarising EAs); use =sample()[['Region']]=
when you want every surveyed household covered.

** Rotating panel and refreshment sample

Beginning in 2013-14, one-third of the original sample is refreshed
each wave.  Households participate for three consecutive waves, then
rotate out.  This creates two sub-populations within each wave:

- Continuing panel :: Households tracked from a prior wave.  These
  receive a /longitudinal/ (panel) weight.
- Refreshment sample :: Newly drawn households entering the sample.
  These receive a /cross-sectional/ weight but no panel weight (or
  a zero/NaN panel weight).

The =rotate= variable in GSEC1 distinguishes the two groups (=rotate=1=
for continuing panel, =rotate=0= for refreshment) in waves that
have it (2013-14, 2015-16).

*** Weight construction (from BID)

The panel weight construction follows an 8-step procedure
(BID Section 6, available at catalog 1001):

1. Begin with 2005 UNHS base weights (=hmult=: inverse probability
   of selection with EA-level non-response correction).
2. Incorporate probability of sub-selection from UNHS into UNPS
   (varies by stratum: 100% for Kampala, 36--43% for others).
3. Incorporate probability of selection into tracking (for split-offs).
4. Fair-share correction for composition changes (factor of 1/2 for
   households receiving new members from other households).
5. Pool the component weights.
6. Derive attrition-adjusted weights via logistic model on individual
   covariates (gender, age, marital status, education, etc.).
7. Trim extreme weights.
8. Post-stratify to known population totals.

The cross-sectional weight (=wgt_X=, =h_xwgt_W5=) is a positive
weight for all sample and non-sample members of interviewed UNPS
households, including refreshment sample households that have no
panel history.  The panel weight is zero or NaN for non-sample
individuals and refreshment households.

For cross-sectional analysis within a single wave, use =weight=.
For longitudinal/panel analysis across waves, use =panel_weight=
(and restrict to households where it is non-null and positive).
