* Survey Program

The Uganda National Panel Survey (UNPS) is conducted by the Uganda
Bureau of Statistics (UBOS) with support from the World Bank
Living Standards Measurement Study (LSMS).  It builds on the
Uganda National Household Survey (UNHS) 2005/06 baseline.

- World Bank catalog :: https://microdata.worldbank.org/index.php/catalog/1001

* Sampling Design
** Overview

Two-stage stratified cluster sample.  Enumeration Areas (EAs) are
the primary sampling unit (PSU), drawn from the 783 EAs visited in
the UNHS 2005/06.  The UNPS selected 322 EAs covering 3,123
households.

The UNPS uses two visits per year (six months apart) to capture
both agricultural seasons.

** Cluster (PSU) identifier

The cluster identifier (=v= in the library's index convention) also
changed:

| Waves       | Variable                     | Type    | Description                  |
|-------------+------------------------------+---------+------------------------------|
| 2005--2012  | =comm=                       | numeric | Community/EA code            |
| 2013-14     | =h1aq4a=                     | numeric | Parish code (8 digits)       |
| 2015-16     | =parish_code=                | numeric | Parish code                  |
| 2018-19     | =distirct_name= + =parish_name= | string  | =DISTRICT/PARISH= composite  |
| 2019-20     | =district= + =s1aq04a=       | string  | =DISTRICT/PARISH= composite  |

All are drawn from GSEC1 (the household cover page).  The shift
from numeric EA codes to parish names occurred around 2015-16 as
the sampling frame evolved from strict EA boundaries to parish-level
clusters to accommodate split-off households.

*** Why 2018-19 / 2019-20 =v= is a =DISTRICT/PARISH= composite (GH #323)

Those two waves have no usable parish /code/ (=parish_code= carries only
19 distinct values), so =v= was originally set to the free-text parish
/name/.  **Parish names are not unique in Uganda.**  20 of them (2018-19)
and 23 (2019-20) recur across districts; =CENTRAL= alone appears in TEN
(HOIMA, KAGADI, KALUNGU, KIRYANDONGO, KYOTERA, MITOOMA, ...).  Keying on
the bare name fused genuinely different parishes into a single "cluster",
whose =Region= / =District= / GPS were then resolved by an undeclared
=groupby().first()= — i.e. arbitrarily.

Qualifying the name by its district makes =v= an actual cluster key:
clusters spanning more than one district drop 20 → 0, and the p90
within-cluster GPS spread falls 24.0 → 14.9 km.  Both =sample()= and
=cluster_features()= build =v= from the same pair (see =mapping.v()= in
each wave's =_/=), so the two cannot drift apart.

Deliberately *not* keyed any finer.  Adding =subcounty_name= splits 10
further groups, but every one of those splits is a spelling artefact of a
single subcounty — =NYENGA= / =NYENGA DIVISION= (centroids 0.0 km apart),
=LUBAGA= / =RUBAGA DIVISION= (5.3 km), ="KAGADI  TOWN COUNCIL"= /
="KAGADI TOWN COUNCIL"= (a doubled space) — so it would fragment real
parishes on data-entry noise.

Note the source misspells "district" as =distirct_name= in 2018-19.

*** Hybrid =v= in 2009-10 (movers and split-offs)

The 2009-10 =comm= column is labeled /Current EA Code as in 2005/06
Data/: it is the /original 2005-06/ sampling-frame EA, not a new
2009-10 code.  That convention makes sense for panel continuity
(every /original/ 2005-06 household keeps its EA across waves) but
leaves two groups of 2009-10 households without a usable =comm=:

| =hh_status= | count | why =comm= is blank                                     |
|-------------+-------+---------------------------------------------------------|
| 2 (mover)   |   192 | Left the original EA; no 2005-06 code applies.          |
| 3 (split-off) | 368 | Did not exist in 2005-06; no 2005-06 code applies.     |
| 1 (orig. non-mover, anomaly) | 5 | Isolated data-entry gap (HHIDs =3163001601..1608=). |
| Total       |   565 | 19% of 2 975 interviewed households.                    |

These 565 households are not a data bug — by survey design they are
/outside the 2005-06 sampling frame/ that =comm= encodes.  They do,
however, carry modified GPS coordinates from =UNPS_Geovars_0910.dta=
(=lat_mod=/=lon_mod=, jittered ±2 km urban / ±5 km rural, with a
=source= flag distinguishing current-location from origin-location
fixes), so a within-wave spatial identifier can still be built.

To avoid silent data loss in downstream tables (the old behaviour
dropped these 565 households in =roster_to_characteristics='s
=groupby=), =sample()='s =v= column in 2009-10 is populated by a
hybrid rule:

- =v = comm= (7-digit numeric string) when =comm= is non-blank —
  2 410 households, 317 distinct panel EAs.  Matches CSECTION1's
  =comm= index exactly, so community-questionnaire joins work as
  they always did.
- =v = "@{lat:+.2f},{lon:+.2f}"= (a /synthetic/ label prefixed with
  =@=, binned to a 0.05° grid ≈ 5.5 km) when =comm= is blank but
  coordinates are available — 541 households, ~339 distinct bins.
  These labels do /not/ match any CSECTION1 =comm=, by design: the
  community questionnaire was not administered at the movers' or
  split-offs' 2009-10 locations, so a non-match is the correct
  outcome of a join.
- =v = <NA>= when neither =comm= nor coordinates are available —
  24 households, the refreshment / geo-missing tail that is also
  reported in the =UNPS_Geovars_0910= docs as /Invalid: 24/.

Implementation: in =data_info.yml= the =sample:= block uses the
=dfs:=/=derived:=/=drop:= pattern to merge GSEC1 and the geovars
file, then dispatches to the =coalesce_coord_bin= transformer in
=lsms_library/transformations.py=.  The 0.05° grid resolution and
=@= prefix are documented defaults on the transformer and can be
overridden in the YAML if another wave or country needs different
values.

Why only 2009-10 gets the hybrid treatment: 2010-11 and 2011-12
also use =comm= from GSEC1, but their raw surveys populated it for
every household (movers and split-offs got re-coded with their
original-sample EA), so no fallback is needed.  From 2013-14 onward,
=v= is a parish identifier — every household has one — and the
issue disappears by construction.

*** HH-level =Region= on =sample()=

Since =v= can be a synthetic =@lat,lon= label (§above), consumers that
needed =Region= used to pull it from =cluster_features()= and silently
lose those households.  =sample()= exposes =Region= directly as a
household-level column for every Uganda wave (it reads the same
=region= field that =cluster_features()= does, with the same Kampala
→ Central fold in 2009-10 and 2010-11 where applicable).

Note (GH #323): =cluster_features()= *does* now carry the synthetic
=@lat,lon= clusters.  It previously did not — =sample()= built them via
its =derived: coalesce_coord_bin= block but =cluster_features()= kept the
raw blank =comm=, so all 565 out-of-frame households collapsed away and
**339 =(t, v)= pairs present in =sample()= had no =cluster_features()=
row at all**.  The 2009-10 =cluster_features= hook now invokes the *same*
=fill_v_with_coord_bin= transformer, on the same coordinates with the same
defaults, so the labels agree by construction; the orphan count is 0.  The
24 households with neither a =comm= nor a GPS fix keep =v = <NA>= and are
dropped — nothing can locate them, and a fabricated cluster would be worse
than a missing one.

Practical consequence: =Country('Uganda').sample()[['v','Region']]=
is complete for every household; joining =Region= from =sample()=
— rather than =cluster_features()= — avoids dropping the 565
movers/split-offs from any =(i, t, m)=-keyed regression where =m=
is derived from Region.  Recommended pattern:

#+begin_src python
uga = ll.Country('Uganda')
d = uga.household_characteristics()
# Region = market at HH level, available for all HH including movers
d = d.join(uga.sample()[['Region']])
d = d.rename(columns={'Region': 'm'}).set_index('m', append=True)
#+end_src

Continue to use =cluster_features()[['Region']]= when you want the
cluster-level view (e.g., summarising EAs); use =sample()[['Region']]=
when you want every surveyed household covered.

** Rotating panel and refreshment sample

Beginning in 2013-14, one-third of the original sample is refreshed
each wave.  Households participate for three consecutive waves, then
rotate out.  This creates two sub-populations within each wave:

- Continuing panel :: Households tracked from a prior wave.  These
  receive a /longitudinal/ (panel) weight.
- Refreshment sample :: Newly drawn households entering the sample.
  These receive a /cross-sectional/ weight but no panel weight (or
  a zero/NaN panel weight).

The =rotate= variable in GSEC1 distinguishes the two groups (=rotate=1=
for continuing panel, =rotate=0= for refreshment) in waves that
have it (2013-14, 2015-16).

*** Weight construction (from BID)

The panel weight construction follows an 8-step procedure
(BID Section 6, available at catalog 1001):

1. Begin with 2005 UNHS base weights (=hmult=: inverse probability
   of selection with EA-level non-response correction).
2. Incorporate probability of sub-selection from UNHS into UNPS
   (varies by stratum: 100% for Kampala, 36--43% for others).
3. Incorporate probability of selection into tracking (for split-offs).
4. Fair-share correction for composition changes (factor of 1/2 for
   households receiving new members from other households).
5. Pool the component weights.
6. Derive attrition-adjusted weights via logistic model on individual
   covariates (gender, age, marital status, education, etc.).
7. Trim extreme weights.
8. Post-stratify to known population totals.

The cross-sectional weight (=wgt_X=, =h_xwgt_W5=) is a positive
weight for all sample and non-sample members of interviewed UNPS
households, including refreshment sample households that have no
panel history.  The panel weight is zero or NaN for non-sample
individuals and refreshment households.

For cross-sectional analysis within a single wave, use =weight=.
For longitudinal/panel analysis across waves, use =panel_weight=
(and restrict to households where it is non-null and positive).

* Unit handling for =food_acquired=

The =u= (unit) index level on =food_acquired= is canonical at
parquet-write time via =harmonized_unit_labels()= in
=lsms_library/countries/Uganda/_/uganda.py=.  It reads the =#+name: u=
table in =_/categorical_mapping.org= via =get_categorical_mapping=
(Tier 1 convention shared with Malawi, Mali, Nigeria, Senegal, Burkina
Faso) and returns a =Code -> Preferred Label= dict where the canonical
labels (=Kg=, =Gram=, =Litre=, =Akendo (Small)=, =Sack (120 kgs)=,
...) are already in human-readable form.  =food_acquired()= renames
the =u= index level through this map, so by the time the wave parquet
is written the =u= index carries clean labels.

The =u= table also retains the per-wave variant columns (=2005-06=,
=2009-10=, ..., =2019-20=) that document the raw label appearing in
each wave's questionnaire.  These extra columns are unused at runtime
but preserve the cross-wave provenance previously captured in the
retired =_/unitlabels.py= file.

The legacy =_/unitlabels.csv= /=_/unitlabels.py= pipeline (which
emitted a 2-column projection of the same data) was retired in
favour of the org-table single source of truth.

The =_/conversion_to_kgs.json= file is a kg-per-unit factor
lookup consumed by =food_quantities= derivation -- a separate
concern from =u= label canonicalisation.  Its keys are the same
=Preferred Label= strings used above; it is built dynamically by
=_/kg_per_other_units.py=.

For the cross-country canonical-=u= roadmap, see GH #223.

* =people_last7days= counts MEMBERS, not visitors

The question ("how many people were present in the last 7 days?") is asked
separately for household /members/ and for /visitors/.  Through 2013-14 the
two live in parallel COLUMNS and the library maps =Men/Women/Boys/Girls= to
the member block, ignoring the visitor block:

| Wave    | File     | Member columns (used) | Visitor columns (ignored) |
|---------+----------+-----------------------+---------------------------|
| 2005-06 | GSEC14   | =h14q1--4=            | =h14q5--8=                |
| 2009-10 | GSEC15A  | =h15a1--4=            | =h15a5--8=                |
| 2013-14 | GSEC15   | =T6FQ01a--d=          | =T6FQ01e--h=              |

From 2018-19 the questionnaire went LONG: the same distinction moved from a
column suffix into a ROW category.  =GSEC15A= now has two rows per household
keyed by =CEA01= ∈ {=Household members=, =Visitors=}, and =CEA01A--D= are the
counts /for the selected category/.  The extraction was never updated, so both
rows collided on the declared =(i, t)= index and =groupby().first()= kept
whichever the file happened to list first — effectively a coin flip.  ~49% of
households therefore carried VISITOR counts, and ~1,985 came back with ZERO
people (GH #323).

Fixed with a =where:= row filter in each wave's =data_info.yml=, selecting the
member rows and thereby making =(i, t)= genuinely unique, so the collapse never
fires.  Mean people/household: 2.86 → 4.60 (2018-19), 2.78 → 4.55 (2019-20).

If you want "people fed" = members + visitors, that is a DIFFERENT table and
should be declared as one; the five-wave precedent above is members-only.

* =cluster_features= aggregation policy and its =<NA>= cells

=cluster_features= is built by reducing HOUSEHOLD rows to cluster grain
(=final_index: [t, v]=, GH #161).  That reduction is intended, but until GH #323
it ran through an undeclared =groupby().first()=.  It is now declared in
=_/data_scheme.yml= under =aggregation:= and enforced by the framework:

- =Region= / =District= / =Rural= → =unique=: the value if every member
  household agrees, else =<NA>= plus a warning.
- =Latitude= / =Longitude= → =median=: a robust centroid (a no-op on clean
  clusters, whose within-=v= spread is 0.0 km, and resistant to the occasional
  wild geovar coordinate that would drag a =mean= across the country).

**Consumers will now see =<NA>= where they previously saw a value.**  That is
the point: the old value was an arbitrary pick among disagreeing households.
Two legitimate causes of disagreement remain, and neither is a bug:

1. /Panel dispersal/ — =comm= (2005-06 … 2011-12) is the *2005-06 EA of
   origin*, so a household that MOVED still carries it while now living in
   another district.  93 (2010-11) and 106 (2011-12) clusters are affected.
   Their district is genuinely undetermined from this table alone.  (=mode= was
   tried and rejected: validated against the 2005-06 frame it recovered the
   right district for only 78% of the ambiguous groups — and just 84% of the
   /unambiguous/ control groups, because Uganda split its districts between 2005
   and 2011.)
2. /Sub-cluster variation/ — =Rural= is an EA attribute, but =v= is a PARISH
   from 2013-14 on, and a parish contains both urban and rural EAs.  So =Rural=
   simply is not a parish-level attribute (70 / 65 / 122 / 122 clusters in
   2013-14 / 2015-16 / 2018-19 / 2019-20).

For a per-household answer, use =sample()=, which carries exact household-level
=Rural= and =Region= for every household.
