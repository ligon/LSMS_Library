#+TITLE: Côte d'Ivoire (CotedIvoire) — LSMS Library Notes
#+DATE: 2026-04-12

* Survey Overview

Côte d'Ivoire has five survey waves in the LSMS Library spanning 1985–2019:

| Wave    | Survey Program                  | Catalog ID | Format   | Status                                  |
|---------+---------------------------------+------------+----------+-----------------------------------------|
| 1985-86 | CILSS / EPAM (LSMS wave 1)      | 83         | DAT/DCT  | sample, cluster_features, roster        |
| 1986-87 | CILSS / EPAM (LSMS wave 2)      | 84         | DAT/DCT  | sample, cluster_features, roster        |
| 1987-88 | CILSS / EPAM (LSMS wave 3)      | 85         | DAT/DCT  | sample, cluster_features, roster        |
| 1988-89 | CILSS / EPAM (LSMS wave 4)      | 82         | DAT/DCT  | sample, cluster_features, roster        |
| 2018-19 | EHCVM (Enquête Harmonisée)      | 4292       | Stata DTA| Fully integrated                        |

* 2018-19 Wave (EHCVM)

This wave follows the standard EHCVM pattern used by Togo, Benin, Guinea-Bissau, and other
West African countries. Variable naming conventions are identical to other EHCVM countries.

** Filename Convention

CotedIvoire 2018-19 uses uppercase =CIV= in filenames (e.g., =s00_me_CIV2018.dta=),
unlike most EHCVM countries which use lowercase (=ben=, =tgo=, =gnb=, etc.).

** DVC-Tracked Files

The following EHCVM aggregate files are available from the WB catalog (4292) but are NOT
yet DVC-tracked in this repository:
- =ehcvm_individu_CIV2018.dta= (individual characteristics; 61,116 cases, 53 vars)
- =ehcvm_ponderations_CIV2018.dta= (sampling weights; 1,084 cases, 2 vars)
- =grappe_gps_CIV2018.dta= (anonymized GPS; 1,084 cases, 7 vars)

These files are downloaded via the WB NADA API on demand when =get_dataframe()= is called.
To add them to DVC: run =lsms_library.data_access.push_to_cache_batch()= after downloading.

** Sampling Design

The EHCVM 2018-19 is a single-round cross-sectional survey (not a panel). There are
two vagues (rounds) within the survey:
- Vague 1: dry season
- Vague 2: rainy season

Households are assigned to exactly one vague (=vague= column in s00). Following the
EHCVM standard (CLAUDE.md note on EHCVM countries):
- =v= = =grappe= (PSU/cluster; single variable, NOT compound =vague + grappe=)
- =i= = =[grappe, menage]= (NOT =vague + grappe + menage=)

Weights are at the cluster level in =ehcvm_ponderations_CIV2018.dta= (variable
=hhweight=). Since this is cross-sectional only (no prior panel), =weight= and
=panel_weight= both point to =hhweight=.

Strata are identified via =s00q01= (Region) combined with =s00q04= (Urban/Rural).

** household_roster Note

The =ehcvm_individu_CIV2018.dta= file (derived aggregate) was not available in DVC at
bootstrap time. The =s01_me_CIV2018.dta= file (raw Section 1 responses) is used instead
for =household_roster=. It contains the same household members (61,116 cases) with
variables =s01q01= (sex), =s01q04a= (age), =s01q02= (relationship). The person ID is
constructed as =[grappe, menage, s01q00a]=.

For =individual_education=, the =ehcvm_individu_CIV2018.dta= file is still referenced
because it contains the harmonized =educ_hi= variable. This file must be downloaded via
WB API; it is not DVC-cached.

* 1985-89 Waves (CILSS EPAM)

** Implementation Status (2026-04-12)

The =.DAT/.DCT= files are readable by =get_dataframe()= from =local_tools=. The prior
agent incorrectly concluded they were not. Three features are now YAML-path implemented:

| Feature          | Source File          | Notes                                   |
|------------------+----------------------+-----------------------------------------|
| sample           | WEIGHT{85,86,87,88}  | i=[CLUST,NH], v=CLUST, Rural from REGION|
| cluster_features | WEIGHT{85,86,87,88}  | 100 clusters, Region + Rural columns    |
| household_roster | F01A / SEC01A        | SEX, AGEY, REL (codes 1-13 mapped)      |

Note: The 1988-89 wave uses =SEC0*= prefix instead of =F0*= prefix for all files.
The roster file is =SEC01A.DAT= (same variable names as F01A).

** Household ID Convention

All 1985-89 waves use =i = [CLUST, NH]= formatted via the =cotedivoire.i()= function
(CLUST zero-padded + NH zero-padded to 3 digits). This matches the existing
=2018-19= convention for the =i= formatter.

** Sampling Design

- 100 clusters (PSUs), ~16 households per cluster
- 5 regions: ABIDJAN, OTHER CITIES, EAST FOREST, WEST FOREST, SAVANNA
- Urban = ABIDJAN + OTHER CITIES; Rural = EAST/WEST FOREST + SAVANNA
- Weight variable: ALLWAITN (normalized all-household weight)
- Panels: these are longitudinal (same households followed across 4 waves)
  but household IDs may change — no panel_ids implemented yet

** Key Variable Mapping (1985-89 roster)

REL codes (F01A / SEC01A):
| Code | Label                      | Kinship tuple              |
|------+----------------------------+----------------------------|
|    1 | Head                       | [0, 0, consanguineal]      |

|    5 | Grandchild                 | [-2, 0, consanguineal]     |
|    6 | Parent Of Head Or Spouse   | [1, 0, consanguineal]      |
|    7 | Sibling                    | [0, 1, consanguineal]      |

** What Is NOT Available for 1985-89

- GPS coordinates (Latitude/Longitude): predates GPS surveys
- Housing (Roof/Floor): F02A has dwelling type (DOMICILE) but not material
- interview_date: date split across DAY1/MO1/YR1 — YAML path can't construct datetime
  (would need a Python script; not yet implemented)
- shocks, assets, food_acquired: not attempted yet (complex multi-file modules)

** Legacy food_expenditures

Wave-level =_/food_expenditures.py= scripts exist for 1985-86, 1986-87 using the
=cotedivoire.food_expenditures()= function from =_/cotedivoire.py=. These read =F12A.DAT=
files as CSV format via =dvc.api.open=. They are legacy scripts and use the retired
=lsms.tools= package — do not extend them.

** Known source defects (GH #323)

*** cluster_features is projected from a HOUSEHOLD-grain source

All five waves read =cluster_features= from a household-grain file
(=WEIGHT{85..88}.DAT=; =Menage/s00_me_CIV2018.dta= — the same files =sample=
reads per household) while the table's grain is one row per cluster =(t, v)=.
=cotedivoire.cluster_features()= projects to cluster grain in the extraction.
Every declared column is a cluster-CONSTANT attribute, so the projection loses
nothing; before it existed, the framework silently collapsed 17,896 surplus
household rows with =groupby().first()=.

Where a cluster's households DISAGREE about a cluster attribute the projection
takes the *strict majority* and warns (naming the cluster and the competing
values); on a tie it emits =<NA>= rather than let source row order decide.
One real conflict exists:

- *grappe 648* (CAVALLY, 2018-19): 11 households coded =Rural=, 1 coded
  =Urbain= (distinct raw labels, not a case variant). Resolves to =Rural=.

*** CLUST 122 has corrupted person keys (1988-89)

=SEC01A.DAT= repeats =(CLUST, NH, PID)= on 4 rows, all in cluster 122; the
declared index is =(t, i, pid)=, so the framework silently deleted them.
Handled in =1988-89/_/mapping.py= (wave-scoped on purpose — a country-wide dedup
rule would hide the next occurrence), which WARNS on every collision:

- =(122, 19, PID 2)= — same F, REL 10, age *46 vs 45*. One person, double-entered.
  Collapsed deliberately; the discarded age is named in the warning.
- =(122, 21, PID 10)= — =SEX=F= and =SEX=M=, both age 10, both Son/Daughter.
  The sex disagrees, so these two rows cannot be one person's consistent record.
  *Both reported records are kept*, re-keyed =10= / =10_2=.

  *UNRESOLVED AMBIGUITY, stated plainly*: the source does not say whether this
  household has two 10-year-olds who were given one PID (plausible — it is a
  polygamous household with two spouses, and it already contains two same-age
  different-sex children as separate pids: 11=F,8 and 12=M,8), or ONE child
  whose sex was corrupted on one of two entries. =HHEXP88.DAT.HHSIZE8= is 13 for
  this household (= the roster ROW count), but it is *not* independent evidence:
  it is itself built by counting rows of this same defective file, so it counts
  the duplicate either way. Keeping both records invents no *measurement* (every
  Sex/Age/Relationship emitted is as reported; only the key label is synthetic),
  whereas collapsing would assert a Sex with a coin-flip chance of being wrong
  AND delete a reported person. If a maintainer prefers to drop the ambiguous
  person loudly instead, that is a one-line change in
  =_resolve_pid_collisions=.

- =individual_education= duplicates pids 1 and 2 in the same household (122, 21);
  those pairs agree on the emitted column and are collapsed as double-entries.

Separately (pre-existing, NOT #323): =(CLUST 137, NH 6, PID 12)= is a *blank*
roster line (Sex =<NA>=, Age ='.', Relationship =<NA>=) and is dropped by
=_finalize_result=. So the 1988-89 roster is 10,563 reported records − 1
double-entry − 1 blank = *10,561* at the API.

*** harmonize_seed_crop must stay INJECTIVE

=plot_inputs= indexes on =(t, i, input, crop, u)= and =harmonize_input= maps every
seed label to the single input =Seed=, so two seed labels sharing a =crop=
Preferred Label collide on one key and one is silently dropped. A four-label
=Autre crop= catch-all was doing exactly that, discarding 9 reported rows. The
buckets are now split; the guard is the uniqueness assertion at the end of
=2018-19/_/plot_inputs.py= — see the note in =_/categorical_mapping.org=.

* Household Presence / MonthsSpent

The EHCVM questionnaire (used in the 2018-19 wave) does NOT ask
"How many months did [NAME] live here in the past 12 months?"
(the continuous MonthsSpent/MonthsAway variable available in Uganda,
Ethiopia, Tanzania, Malawi, and many other LSMS countries).  Instead,
EHCVM uses two binary eligibility filters:

- =s01q12= :: "Has [NAME] lived continuously in this household for at
  least 6 months?"
- =s01q13= :: "Does [NAME] intend to stay at least 6 months?"

These determine membership eligibility but do not give a continuous
duration measure.  Downstream analysts who need a presence filter for
=household_characteristics= should use these binary variables directly
rather than expecting a =MonthsSpent= column in =household_roster()=.

The pre-EHCVM 1985-89 waves (CILSS/EPAM) also lack a continuous
months-of-presence variable; these surveys predate the EHCVM design
and do not include MonthsSpent or equivalent fields.

* Unit handling for =food_acquired=

The =u= (unit) index level on =food_acquired= carries clean canonical
French unit names (=Kg=, =Litre=, =Sachet=, =Unité=, =Cuillière=,
=Régime=, =Boîte de tomate=, ...) -- 52 distinct values total.  No
=u= table is wired into =_/categorical_mapping.org=; the framework's
auto-discovery on =u= would be a no-op (the labels are already
canonical).

The clean labels come from a per-wave mojibake fix in
=2018-19/_/mapping.py=:

- The .dta file's value-labels block stores French strings as raw
  UTF-8 bytes, but its metadata declares Latin-1 encoding.  pyreadstat
  respects the metadata and produces double-decoded strings (=Unité=
  becomes =UnitÃ©=, =Cuillière= becomes =CuilliÃ¨re=, etc.).
- The wave's =mapping.py= defines a =food_acquired= post-processor
  (replacing the default =food_acquired_to_canonical= alias) that calls
  the canonical reshape and then sweeps mojibake out of every string
  column / index level via =_decode_mojibake_in_df()=.  See the
  module's docstring for the encoding analysis.
- No lost-prefix substitutions are needed for CotedIvoire 2018-19:
  every distinct =u= and =j= value cleans up via the byte-level
  decode alone.  =_LOST_PREFIX_REPLACEMENTS= is therefore an empty
  dict (kept for parity with the GuineaBissau / Niger
  implementations).

After this fix landed:
- =food_acquired()= index level =u=: 52/52 clean (was 47/52).
- =food_acquired()= index level =j=: 136/136 clean (was 90/136).

Cache caveat: anyone with cached parquets predating this commit must
run =lsms-library cache clear --country CotedIvoire= to pick up the
fix; the staleness check does not see =mapping.py= edits.

For the cross-country canonical-=u= roadmap, see GH #223.  CotedIvoire
was originally classified Tier 3 (needing a new =u= table built) in
that issue's inventory, but turned out to be Tier 2 once the mojibake
fix landed -- the underlying labels are already canonical.
