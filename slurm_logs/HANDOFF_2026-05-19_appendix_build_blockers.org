parallel scatter-gather across waves.  The coordinator <-> worker
channel ran through =slurm_logs/build_2026-05-18/{MESSAGES,QUESTIONS}_*=,
which the parent skill documents as "ephemeral, safe to rm".  Most of

used (empty files) and are not preserved.

** 1995 worker -- shared formatter NaN crash

#+begin_quote
food3.dta has exactly 1 row (row idx 6121, q1c=9) where q2 (within-village
HH number) is NaN while it still carries food data (item 16, purch qty 2).
The shared =_/ethiopiarhs.py:i= composite-id formatter does
  =tools.format_id(...) + '_' + tools.format_id(value.iloc[1])=
and format_id(NaN)->None, so c.food_acquired() raises
  TypeError: can only concatenate str (not "NoneType") to str
during the 1995 build.  1994a/food1 and 1994b/food2 had ZERO NaN keys so
never exercised this.  Editing =_/ethiopiarhs.py= is OUT OF SCOPE; YAML-path
extraction (no script / no df_edit allowed for food per recipe) cannot
drop the malformed row from within 1995/.  Proceeding to check for an
in-scope framework mechanism (e.g. idxvars NaN handling); if none exists
this wave needs either (a) a one-line NaN-guard in the shared =i=
formatter or (b) recipe permission for a 1995-local df_edit.
#+end_quote

*Resolution* (coordinator-side, 2026-05-18): option (a).  The shared
=_/ethiopiarhs.py:i= formatter NaN-guards to return =pd.NA=; the food
melt drops NA-=i= rows.  Workers told to NOT work around it themselves
and NOT to edit =_/ethiopiarhs.py= (see /coordinator broadcast 1/
below).

** 1989 roster worker -- pre-expansion instrument divergence

#+begin_quote
STOP+REPORT: 1989 is the pre-expansion instrument and its code schemes
DIVERGE from the 1994+ shared tables.  Roster source chosen:
=demog89_1.dta= (ID 2301923) -- 2302 rows, ONE row per individual, has
site/hhid/pid/sex/age/relnhhed.  =demog89_2.dta= (ID 2301927) is a
203-row HH-level summary, NOT a roster -> rejected.

BLOCKER 1 -- relation codes differ (cannot reuse =erhs_relation=):
  1989 relnhhed value labels:
    1 Household Head | 2 Wife | 3 Own child | 4 Adopted child
    5 Illegitimate child | 6 Brother/sister | 7 Mother/father
    8 Helper | 9 Other  (+ stray codes 11 x1, 36 x1, NaN x3)
  Only codes 1,2 align loosely; 4,5,7,8,9 MEAN DIFFERENT THINGS.

BLOCKER 2 -- composite =i= cannot be expressed in pure YAML:
  1989 demog89_1 has NO separate village column + NO separate hh_no
  column.  Identity is packed into one 5-digit hhid = R(1)VV(2)HH(2)
  [region * region+PA + within-PA HH no], e.g. 10101 = region 1, PA 01,
  HH 01.  Pure-YAML =idxvars= cannot derive two columns from one.

Awaiting coordinator decision: (a) add =erhs_relation_1989= + 1989
village/hh split helper to the shared files yourselves, or (b)
authorize a 1989-local wave script/df_edit for the hhid split and a
1989 relation mapping.
#+end_quote

*Resolution* (coordinator-side, post-1990s batch): option (a) for
both blockers.

- New =erhs_relation_1989= table in
  =lsms_library/countries/EthiopiaRHS/_/categorical_mapping.org=
  mapping 1989's 9-code =relnhhed= taxonomy to Preferred Labels that
  already resolve in =lsms_library/categorical_mapping/kinship.yml=.
  Judgment calls recorded in the table's preamble: code 5
  "Illegitimate child" -> child (consanguineal); code 8 "Helper" ->
  Servants; code 9 "Other" -> Other unrelated person.  Stray data
  errors (codes 11 x1, 36 x1) map to NA.
- Shared =_/ethiopiarhs.py= grew a *scalar* =i= path alongside the
  composite (village, hh_no) path: the packed 5-digit =hhid= is
  carried through as a single scalar identifier.  No 1989 wave script
  / df_edit needed.
- Caveat documented in =EthiopiaRHS/_/CONTENTS.org=: =demog89_1= is a
  partial person roster (338 HH) vs =food89= (483 HH; 279 shared) --
  1989 =food_expenditures= is meaningful only for the ~279 shared HH;
  no fuller 1989 person roster exists in the archive (=demog89_2= is a
  203-row HH-level summary).

* Lessons for the next ERHS batch

- The "STOP+report on shared-file changes" stop-list works.  Workers
  refused to silently invent =erhs_relation_1989= or hack the shared
  formatter; the coordinator made the policy choice (option (a) both
  times) and the worker re-verified against the new shared state.
- The single-row =food3.dta= NaN-key bug was lurking behind a
  YAML-path extraction that had been green on every other wave;
  cross-wave verification surfaces this class of bug, per-wave
  verification never will.  Keep the all-waves =is_this_feature_sane=
  step in the recipe.
- The 1999 / 2004 / 2009 STOP+report (from the same post-1990s
  dispatch, not preserved here) led to the wired/unwired split
  documented in =EthiopiaRHS/_/CONTENTS.org= (no person roster + no
  item-level food in the Dataverse archive for R5-R7; only
  pre-aggregated consumption scalars; declared as waves but unwired).
  PR #276 closes the fresh-checkout =FileNotFoundError= that this
  stub-wave layout caused.
