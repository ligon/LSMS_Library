* Purpose

Uganda landed as the Phase-1 pilot (PR #280, all 8 UNPS waves, 42,841
plot records).  This doc is the *repeatable workflow* for rolling
~plot_features~ out to the remaining 12 LSMS-ISA countries named in GH
#167.  It exists so that (a) the design is reviewable before any
implementation agent is dispatched, and (b) every per-country
implementation shares one canonical reference instead of 12 divergent
prompt copies.

The governing principle (scrum-master skill): *delegate the search and
the mechanical implementation; keep the schema/vocabulary synthesis.*
The shape of a per-country implementation is identical; the *content*
(source files, column names, unit conventions, code vocabularies)
varies and must be discovered, not guessed.

* The Uganda pattern (what every country reproduces)

Five artifacts per country, all under ~lsms_library/countries/<C>/~:

1. *Per-wave extraction script* ~<C>/<wave>/_/plot_features.py~ — loads
   the wave's agricultural-module source file(s) with
   ~get_dataframe(..., convert_categoricals=False)~, defines a per-wave
   column-name map (~colmap~), calls the country helper, asserts a
   unique ~(t,i,plot_id)~ index and >0 rows, writes ~plot_features.parquet~.
2. *Shared per-wave harmonizer* ~<C>/_/<country>.py:plot_features_for_wave~
   — the harmonization logic: area→hectares, raw codes→canonical
   vocab via ~categorical_mapping.org~ tables, builds the canonical
   ~(t,i,plot_id)~ frame with columns Area, AreaUnit, Tenure,
   TenureSystem, SoilType, Irrigated.
3. *Country-level concatenator* ~<C>/_/plot_features.py~ — concat the
   per-wave parquets, ~id_walk~ to the panel canonical id scheme, write
   ~var/plot_features.parquet~.
4. *Categorical tables* appended to ~<C>/_/categorical_mapping.org~:
   ~harmonize_tenure_system~, ~harmonize_soil~, ~harmonize_water~,
   ~harmonize_acquire~ (names/values are country-specific).
5. *data_scheme.yml* gains a ~plot_features:~ block declaring index
   ~(t, i, plot_id)~ and a country note.

Note: ~plot_features~, ~food_expenditures~ etc. are auto-surfaced; the
table is NOT a "derived" table — it has real wave-level scripts, so it
*is* declared in ~data_scheme.yml~ (unlike the auto-derived food/roster
tables).

Reference implementation to read before writing any prompt:
- ~lsms_library/countries/Uganda/2013-14/_/plot_features.py~ (richest wave)
- ~lsms_library/countries/Uganda/_/uganda.py:551~ (~plot_features_for_wave~)
- ~lsms_library/countries/Uganda/_/plot_features.py~ (concatenator)
- ~lsms_library/data_info.yml:103~ (canonical Columns + spellings)

* Adversarial verification (cross-cutting — gates every phase)

Plausible-but-wrong is the dominant failure mode here: a recon agent
names the wrong area column, a harmonize_* table omits a code so rows
silently go NaN, a unit factor is inverted.  None of these throw — they
produce a table that builds clean and is quietly corrupt.  So every
phase is paired with an independent skeptic whose mandate is to REFUTE,
defaulting to "something is wrong; prove it."

- *A.0 Pilot audit (done 2026-06-03)*: adversarial auditor poking the
  landed Uganda data BEFORE the pattern is cloned 11×.  Attack list:
  silent unmapped-code drops (the prime suspect behind 56% SoilType
  NaN), area-conversion direction/sanity, household-id orphans vs
  sample(), silent row loss, plot_id collisions masked by the _A/_B
  suffix, tenure-default content-emptiness, wave×column populated
  matrix.  Findings feed back into the canonical pattern before fan-out.
- *A.5 Recipe refutation*: as each recon recipe lands, one independent
  verifier tries to refute its load-bearing claims — "is THIS really the
  lasting-attributes file?", "does THIS column mean area?", "is the unit
  factor right and right-way-round?", "do these codes cover the data?".
  A recipe a skeptic cannot break feeds implementation; one it breaks
  goes back to recon with the specific gap.  Diversity over redundancy:
  the verifier re-derives the key facts independently rather than
  re-reading the recon agent's notes.
- *C.5 Post-build audit*: after each country's implementation, run the
  Uganda A.0 attack list against that country's output (unmapped-code
  drops, area sanity, orphan rate, populated matrix) before its PR is
  considered landable.  This is the per-country generalization of A.0.

These are read-only and never edit source; they report quantified
findings (every claim carries a number).  The scrum master arbitrates —
a skeptic's finding is a hypothesis to confirm, not an auto-revert.

* The workflow: four phases

** Phase A — Reconnaissance (DELEGATE, parallel, read-only)

One agent per *recipe* (so: 1 EHCVM + 4 bespoke = 5 recon agents;
GhanaSPS deferred).  Each agent investigates the country's ag module
and returns a structured *extraction recipe* — data, not code.

Recipe schema (the agent's required output):
- Source file(s) per wave holding *lasting* parcel attributes
  (ownership, area, soil, water/irrigation, tenure system).
- ~colmap~ per wave: which raw column → which canonical field
  (hhid, parcel_id, area_gps, area_est, tenure_system, acquire,
  soil_type, water_source).
- Area unit(s) and the hectare conversion factor(s).
- Raw code → canonical label tables for tenure_system / soil / water /
  acquire (the values that will seed the harmonize_* org tables).
- The household-id quirk: which column matches ~sample().i~ vs. an
  internal section index (Uganda's ~HHID~ trap — see 2013-14 colmap note).
- plot_id construction rule (Uganda = parcelID + '_A'/'_B' for the two
  source files; other countries may have a single parcel roster).
- Wave list + rough row-count estimate per wave.
- Whether GPS is decimal-degree (emit Latitude/Longitude) or absent/
  non-standard (defer, like Uganda).
- Build-path verdict: simple YAML-equivalent vs. needs ~materialize:make~
  script (multi-round / pp-ph).

Recon agents do NOT write code and do NOT edit any file.  Output is a
markdown recipe block appended to a per-country section of a shared
~slurm_logs/RECON_plot_features_<cluster>_<date>.md~.

Dispatch mode: ~Explore~ / general-purpose agents, ~run_in_background~,
all in one message (scatter-gather).  Read-only, so the shared checkout
and DVC cache are safe to share.

** Phase B — Schema & vocabulary synthesis (KEEP — scrum master)

Review the 5 recipes.  This is the judgment work that ties downstream
tests to the schema:
- Reconcile each country's tenure / tenure-system / soil vocabularies
  against the canonical ~spellings~ lists in ~data_info.yml~.  *Extend*
  the canonical lists (e.g. add a country's customary regime) rather
  than force-fitting to a foreign label — the canonical block already
  invites this for TenureSystem.
- Decide area-unit conversions; confirm hectare is the canonical Area.
- Decide GPS per country (emit vs. defer).
- Resolve any cross-country label collisions before they reach tests.
- Approve each recipe (or send it back to recon with specific gaps).

Output: the approved recipes become the shared design artifacts the
implementation agents reference.  Any canonical ~data_info.yml~ spelling
extensions are made by the scrum master directly (one small commit on
the integration branch), since they affect the cross-country schema.

** Phase C — Implementation (DELEGATE, worktree-isolated, one agent per country)

One agent per country (7 EHCVM + 4 bespoke).  Each writes the five
artifacts for its country against the *approved recipe*, builds,
verifies, and leaves a committed branch.  The prompt is a tight
implementation brief that *references* the recipe — it does not embed
the design.

Critical repo-specific dispatch rules (from CLAUDE.md + scrum-master skill):
- *Worktree base discipline*: put the self-reset block at the top of
  every worker prompt —
  : git -C "$(pwd)" fetch origin <integration-branch>
  : git -C "$(pwd)" reset --hard origin/<integration-branch>
  : git -C "$(pwd)" log -1 --oneline
  Multi-agent dispatches have been observed landing on stale ~master~.
- *.pth-pinned imports*: the main venv's ~lsms_library.pth~ hardcodes the
  *main* checkout path, so ~import lsms_library~ inside a worktree runs
  main-checkout code regardless of PYTHONPATH.  A worker that rebuilds
  to verify is silently testing the wrong tree.  Mitigation: have each
  worktree agent build a fresh venv (~python -m venv && poetry install~)
  OR verify by static diff only.  Prefer fresh-venv for plot_features
  because the verification (row counts, schema) genuinely needs to run
  the worktree's code.
- *DVC cache is shared* across agents at ~~/.local/share/lsms_library/~ —
  concurrent agents building *different* countries do not conflict.
- *Stop-list* (every worker prompt): do NOT touch ~data_info.yml~
  canonical block (scrum master owns schema), any file matching
  baseline/golden/expected/snapshot, ~pyproject.toml~/lockfiles, or any
  other country.  Do NOT weaken/skip tests.  STOP + report if the task
  seems to need any of these.
- *Mandatory report format*: lead with "SCOPE DEVIATIONS" (files
  touched outside the country dir); then commit SHA + branch,
  verification output (one line per command incl. row count +
  ~plot_features()~ shape), surprises, worktree path.
- *Trust but verify*: after each worker, run ~git status~ on the *main*
  checkout and ~git -C <worktree> status~ — catch silent main-checkout
  edits regardless of the self-report.

Verification each agent must run (in its own venv):
: Country('<C>').plot_features()   # shape, index names, wave coverage, NaN frac

** Phase D — Cross-country verification (KEEP + delegate the compute)

After implementations land on the integration branch:
- ~Feature('plot_features')()~ across all done countries — confirm the
  ~country~ index level prepends and the union schema is consistent.
- Run ~tests/test_schema_consistency.py~ + ~tests/test_feature.py~
  (the ~test_column_table_has_countries~ guard now passes for
  plot_features since ≥1 country declares it).
- Sanity checks: Area in plausible hectare range per country, no
  duplicate ~(country, t, i, plot_id)~, per-country NaN-fraction table.
- Delegate the full ~--rebuild-caches~ pytest run to a Slurm job with a
  Monitor filter (deterministic process; one-way log stream).

* Branch / PR strategy

- Cut an integration branch ~feature/plot-features-167~ off
  ~development~.  Each country implementation is its own PR *based on
  the integration branch* (or merged into it), so countries land
  independently and a broken country doesn't block the rest.

  risk once the shared recipe is approved), then the 4 bespoke ISA
  countries, then GhanaSPS bootstrap.
- Final integration → ~development~ → ~master~ when coverage is complete
  and ~Feature('plot_features')()~ is green cross-country.
