name.  Our 3.67.0 silently accepts and drops =cache_remote_stream=

* Lessons learned

** Empirical verification beats design doc trust

The parallel agent's design doc was confidently specific about
the fix (=cache_remote_stream=True=), cited a PR (iterative/dvc#9183),
and laid out a clean diagnostic recipe.  Almost everything in it
was right *except the actual fix*.  When I read the installed DVC
3.67.0 source after the empirical disproof, the kwarg name was
different (=cache= not =cache_remote_stream=), and even the
correct kwarg name didn't trigger the cache write the doc claimed.
The doc was working from the PR text, not from the running
version.

Always:
1. Read the actual installed library source for the version pinned
   in =pyproject.toml=, not the upstream PR description.
2. After implementing a fix, run an empirical test that observes
   the *side effect* the fix is supposed to produce, not just the
   surface behavior.  Our unit tests passed because they verified
   the kwarg was passed, not that DVC honored it.

** "Brute force is fine" was the right call

Ligon's "we could just delete the cache like brutes" was
methodologically perfect.  Without it I would have shipped the
broken Change 1 without noticing.  The cost was about an hour of
wasted work; the benefit was preventing a no-op fix from sitting
in master indefinitely.

** Mirror reset is high-value when working tree gets crufty

The =mv LSMS_Library LSMS_Library- && git clone LSMS_Library- LSMS_Library=
pattern took about 5 minutes including .venv migration and ate ZERO
risk because the old state was preserved as =LSMS_Library-=.  Saved
multiple debug detours that would otherwise have come from stale
in-tree parquets, polluted ISSUES.md, and accidental ancillary
state interactions.

** Tests that mock the boundary they're testing are worse than useless

The two =test_layer1_*= tests pinned that =DVCFS.open= was called
with =cache_remote_stream=True=.  They passed.  But DVC silently
dropped the kwarg, so the tests pinned a contract the dependency
violated.  The tests gave false confidence.

When the assertion is "we passed kwarg X to method Y", the test is
testing the test harness, not the system.  The right test is
either an integration test against the real dependency, or
verification that the *side effect* of the call matches expectation.

** Background dispatch saves wall time AND keeps the shell responsive

Ligon's "we could be dispatching these benchmark jobs, I suppose"
was a fair criticism of running both Niger and Uganda benches
sequentially in foreground.  After the prompt, every subsequent
expensive bench / probe / verification run went background, and
the scrum master shell stayed available for judgment work while
the compute happened in parallel.

** lru_cache cross-test contamination is a real test smell

=test_stale_cache_triggers_rebuild= was passing pre-session because
=test_trust_cache_short_circuit= happened to populate
=_load_rejected_column_spellings='s lru_cache with real package
data.  The second test relied on the first to set up its mock
state.  When my v0.7.0 fix changed the call ordering, the test
broke.  The fix was to make the second test self-contained by
patching the loaders explicitly -- the same pattern several other
tests in the file already use.  Worth a separate pass to find
other instances of this pattern.
