* Abstract (200 words, placeholder)

We introduce the Berkeley Survey Data (BSD), a Python library
providing a consistent interface to harmonized household microdata
from approximately 25 developing countries, spanning 1987 to 2022
and more than 2 million households.  Unlike prior harmonization
efforts that impose a common schema on heterogeneous surveys
(LIS, IPUMS) or distribute raw data without standardization (the
World Bank Microdata Library), BSD harmonizes the /interface/
rather than the data: users get uniform access patterns across
countries while retaining the ability to drill down to source
variables when their questions require it.

Using BSD, we construct a new set of cross-country purchasing
power parities from household expenditure and implicit price data
--- an alternative to the International Comparison Program (ICP)
PPPs consumed by the Penn World Table.  Because our consumption
PPPs are derived from the same household microdata being deflated,
they are internally consistent with downstream analysis, available
annually rather than at 3-6-year ICP benchmarks, and computable at
sub-national resolution.  We validate the BSD consumption PPPs
against ICP at benchmark years, document systematic divergences,
and demonstrate their use for regional real-income comparisons
that ICP cannot support.  The library is open source and is
distributed as versioned releases for reproducibility.

* Section 1: Introduction (~1500 words)

** Motivation

Cross-country welfare comparisons require household-level data.
Aggregate data (GDP per capita, national accounts) cannot answer
questions about distribution, composition, or heterogeneity of
welfare.  Researchers who want to compare how families in Uganda
and Nigeria respond to food price shocks, or to decompose
cross-country consumption inequality into within- and
between-country components, need microdata from both countries
that can be merged, compared, and analyzed as a single panel.

** The "last-mile" problem in microdata

A researcher today who wants to add a second country to their
analysis faces weeks of work: learning the survey instrument,
decoding value labels, matching variables across waves, building
deflators, handling cluster and weight structures.  This cost
scales linearly with the number of countries.  For a 10-country
analysis, that is easily six months of setup before any substantive
analysis begins.  Most researchers rationally stop at one or two
countries.

** Prior art and its limits

Three existing approaches:

1. *Full harmonization* (LIS, IPUMS).  Produce a dataset in a
   common schema.  Strength: the output file is uniform, and
   cross-country analysis is as simple as a group-by.  Weakness:
   the harmonization process discards information.  Questions that
   require survey-specific detail cannot be answered.

2. *Raw distribution* (WB Microdata Library, country statistical
   agencies).  Make the original files available; leave
   harmonization to the user.  Strength: nothing is lost.
   Weakness: the last-mile problem persists.

3. *Macro aggregation* (PWT, WDI, PovcalNet).  Compute aggregates
   country by country and distribute the results.  Strength:
   immediately comparable.  Weakness: microdata-level analysis is
   impossible.

BSD proposes a fourth approach: harmonize the /interface/ rather
than the data.  Users get uniform access patterns --- every
country's ~household_roster()~ returns a DataFrame with the same
column names and index structure --- while the underlying data
retains its full detail.  When a question requires it, the user
can drill down to the source ~.dta~ file.

** Paper contributions

1. The BSD library and its release 2026.04, covering N countries
   and 2M+ households, distributed under a permissive license with
   reproducible versioning.
2. Architectural principles for interface harmonization, including
   progressive disclosure, sampling-design-based cluster identity,
   and a kinship decomposition that properly represents polygynous
   and blended households.
3. *A microdata-based alternative to ICP/PWT consumption PPPs*
   constructed from household expenditure and implicit price data,
   internally consistent with the microdata being deflated, updated
   annually, and computable at sub-national resolution.  We validate
   against ICP at benchmark years and document the divergences.

The central intellectual argument is the principle of
/harmonizing the interface, not the data/.  When primary sources
are heterogeneous by design, the right response is a uniform
programming interface layered over preserved source detail, not a
forced common schema.  BSD is the first systematic application of
this principle to cross-country household microdata.  We argue
the principle generalizes beyond household surveys to any domain
where comparable analysis across heterogeneous primary sources is
needed.

* Section 3: Architecture (~3000 words --- the technical heart)

** 3.1 Core abstractions

Introduce the ~Country~ class and the ~Feature~ class.  Show a
10-line example that loads Uganda food expenditures and Nigeria
food expenditures and concatenates them.  The reader should see
the uniform API immediately.

** 3.2 Interface vs. data harmonization

Define precisely what BSD does and does not harmonize:

- Uniform: table names, column names, index structure, canonical
  values for constrained columns (e.g. Sex ∈ {M, F}).
- Not uniform: food item categories, measurement units, regional
  administrative boundaries, survey-specific variables outside
  the declared schema.

Describe the layered harmonization:

1. Column renaming (YAML ~data_info.yml~ per wave)
2. Value-level canonical spellings (inverse-dict in the global
   ~data_info.yml~)
3. Categorical mapping tables (org-mode tables per country for
   cross-wave harmonization)
4. Kinship decomposition (global dictionary in ~kinship.yml~)
5. Sampling design join (~sample()~ → ~v~ at API time)

Each layer is transparent to users who need to bypass it.  The
architecture supports drilling down.

** 3.3 The kinship decomposition

The novel technical contribution worth foregrounding.  Traditional
household rosters carry a single ~Relationship~ column (values like
"Head", "Spouse", "Child").  This is fundamentally inadequate:
polygynous households have multiple wives (all "Spouse"), blended
families have foster children and step children collapsed together,
and cross-country comparison of household structure requires finer
distinctions.

Building on Kroeber (1909) and Romney-D'Andrade (1964), we
decompose kinship into four columns:

- ~Sex~: M/F (no longer redundant with Relationship string)
- ~Generation~: vertical distance from head (0=same, +1=parent)
- ~Distance~: collateral distance (0=lineal, 1=sibling, 2=cousin)
- ~Affinity~: consanguineal, affinal, step, foster, unrelated,
  guest, servant

Show examples: polygynous wives all as (0, 0, affinal); foster
children as (-1, 0, foster), distinguishable from step children
(-1, 0, step) and adopted children (-1, 0, consanguineal).

** 3.4 Sampling design as a single source of truth

The ~sample()~ table encodes the survey's sampling design
(household cluster, weight, strata, urban/rural).  Cluster identity
(~v~) is joined from ~sample()~ at API time, not stored redundantly
in feature parquets.  Describe the architectural benefits:
correctness, extensibility, lineage tracking.

Brief aside: this is a good example of the paper-worthy move of
making explicit what every household-survey analyst implicitly does
manually (join cluster identity onto every analysis).

** 3.5 Cross-wave categorical harmonization

org-mode tables in ~categorical_mapping.org~ map variant labels to
canonical forms across waves within a country.  Automatic
application via column-name matching.  Show Uganda food items
across 8 waves as an example.

Important caveat: we explicitly do /not/ harmonize across
countries.  "Maize" in Uganda and "Maize" in Nigeria may or may not
refer to the same preparation; the analyst decides whether to
treat them as the same.

** 3.6 Caching, reproducibility, and versioning

- Data/code separation: parquets live under ~data_root()~, not
  in the repo tree
- Hash-based cache invalidation
- Pinning to a BSD release version:
  #+begin_src python
  import bsd
  uga = bsd.Country('Uganda', version='2026.04')
  #+end_src
- Implications for replication: a paper citing BSD 2026.04 can be
  exactly reproduced five years later

* Section 5: Lead application --- A consumption PPP from household microdata (~4000 words)

The headline substantive contribution.  BSD enables a
microdata-based alternative to the International Comparison
Program's purchasing power parity exchange rates.  Instead of
periodic benchmark price surveys conducted by statistical agencies
every 3-6 years (the ICP approach that PWT consumes), we construct
consumption PPPs directly from the expenditure and implicit price
data in household surveys.

** 5.1 The problem: ICP/PWT PPPs have known limitations

Brief review of ICP/PWT methodology and its pain points:

1. *Infrequent updates*: ICP benchmarks are expensive and happen
   every 3-6 years.  Between benchmarks, PPPs are interpolated or
   held fixed.  This introduces measurement error in cross-country
   real income comparisons at non-benchmark years.

2. *The revision problem*: when PPPs are updated at a new
   benchmark, historical PWT series change.  Papers written
   against PWT 8.0 do not reproduce against PWT 10.0.  The
   Feenstra-Inklaar-Timmer (2015) relaunch paper openly
   acknowledges this.

3. *External basket*: ICP uses a basket of goods and services
   defined externally (by the ICP methodology) rather than derived
   from what households actually purchase.  This matters for
   welfare interpretations.

4. *No sub-national variation*: ICP produces one PPP per country.
   But regional price levels vary enormously within countries
   (urban vs. rural, capital vs. hinterland), and the single
   national PPP obscures this.

5. *Consistency with deflated data*: the PPPs PWT distributes are
   based on a different data generating process than the household
   expenditure data that development economists typically deflate
   with them.  There is no guarantee of internal consistency.

** 5.2 The insight: household surveys already contain prices

Every household survey that records food acquisition captures
/two/ things per purchase: an expenditure (in local currency) and
a quantity (in local units).  The ratio is an implicit price that
the household actually faced in the market, for the specific item
and the specific region, at the specific time of interview.

Aggregating these implicit prices across households within a
country-wave-item cell yields a country-level price index for that
item.  Repeating across all food items gives a country-level
consumption-basket price level.  Comparing country-level price
levels gives a bilateral PPP.  Aggregating bilaterals via standard
multilateral methods (Geary-Khamis, GEKS) gives a full multilateral
PPP matrix.

This is methodologically straightforward /if you have harmonized
cross-country household microdata with unit-converted quantities/.
BSD provides exactly that.  The ~food_acquired~ feature delivers
expenditure and quantity data per household × item × unit; the
unit-conversion layer (kg_per_other_units) converts to a common
metric; the sampling weights from ~sample()~ make the aggregation
population-representative.

** 5.3 Methodology

*** Step 1: Unit values from the microdata

For each household /h/ in country /c/ at wave /t/, and each food
item /j/, compute the implicit price:

$$p_{hcjt} = \frac{\text{expenditure}_{hcjt}}{\text{quantity in kg}_{hcjt}}$$

using BSD's unit-to-kg conversion for non-metric units.  Discard
transactions with zero quantity or zero expenditure.  Winsorize
tails to manage outliers.

*** Step 2: Country-wave-item price level

Aggregate unit values within each (country, wave, item) cell to a
single price, weighted by sampling weights and optionally by
quantity purchased.  Consider the median (robust) vs. the
weighted mean (efficient) trade-off.

*** Step 3: Consumption basket

Construct a representative consumption basket in several ways:

- *Own-basket (Paasche-like)*: each country's basket
- *Common basket (Laspeyres-like)*: world-average shares
- *Geometric average*: Fisher or Törnqvist indices
- *Reference-country basket*: e.g. Uganda as base

Report multiple variants to demonstrate robustness.

*** Step 4: Bilateral PPPs

For each country pair (/c/, /c'/), compute the cost of the
reference basket in each country and take the ratio.  This gives
the bilateral PPP between /c/ and /c'/, directly comparable to
ICP's bilateral PPPs for the same country pair.

*** Step 5: Multilateral PPPs

Apply standard multilateral aggregation (Geary-Khamis or GEKS) to
the bilaterals to produce a full multilateral PPP matrix suitable
for cross-country real income comparison.

*** Step 6: Validation against ICP/PWT

Compare the BSD consumption PPPs with ICP/PWT's published
consumption PPPs at benchmark years.  Interesting questions:
  - Do they agree at benchmark years?
  - Where do they diverge?  Are the divergences systematic (e.g.,
    BSD undervalues rural consumption because households
    substitute toward home production)?
  - How does the BSD series behave at non-benchmark years, when
    ICP is interpolating?

Report correlations, rank correlations, and bilateral
scatterplots.  The analytical story is not that BSD is "right" and
ICP is "wrong" --- it is that BSD provides an /independent and
annually-updated/ alternative that can be used as a consistency
check on ICP and as a primary deflator for microdata-based
analyses.

** 5.4 Results

- *Headline figure*: a scatterplot of log BSD consumption PPP vs.
  log ICP consumption PPP at benchmark years, with 45-degree line.
  A priori I expect these to line up reasonably well but with some
  systematic drift.
- *Table*: bilateral PPPs for N countries in a canonical reference
  year, BSD vs. ICP columns, with ratio.
- *Time series*: plot BSD PPPs at annual/biannual frequency and
  show that between-benchmark drift is non-negligible for several
  countries.
- *Sub-national PPPs*: demonstrate BSD's ability to compute
  region-level PPPs within a country.  Figure: map of real
  consumption per capita by region for 2-3 countries, using BSD
  sub-national PPPs to deflate.  This is something ICP/PWT cannot
  do at all.
- *Decomposition*: of real consumption differences into price
  (PPP) and nominal (income) components, with proper confidence
  intervals.

** 5.5 Discussion

- What this changes: papers that currently deflate African
  household surveys using a single national PPP are implicitly
  assuming no within-country price variation.  BSD provides
  the machinery to relax this assumption.
- Caveats: the food-PPP is not a full-consumption PPP (services,
  rent, durables are underrepresented in LSMS food modules).
  Combine with non-food modules where available.
- Caveats: self-reported quantities have measurement error.  Some
  of the BSD/ICP divergence is attributable to differences in what
  is measured, not differences in what is correct.
- Caveats: aggregation of food items across countries is an
  analytical choice, not a canonical one.  Report robustness to
  different aggregation schemes.
- *The big argument*: for microdata-based welfare analysis, a
  consistent PPP derived from the same data being deflated is
  preferable to an external PPP derived from a different
  measurement system, even if the external one has better coverage
  of non-food consumption.

** 5.6 Relation to other microdata PPP efforts

Short lit review.  The idea is not entirely new --- there have
been country-specific exercises using household surveys to
construct price indices (e.g., Deaton's spatial price indices for
India, Ravallion's regional price adjustments).  The contribution
here is doing this /at scale and consistently/ across ~25
countries, enabled by BSD's harmonized interface.  Cite the
precedents honestly.

* Section 5b (optional): Cross-country CFE demand estimation (~1500 words)

Show the code:
#+begin_src python
import bsd
from cfe.regression import Regression
import numpy as np

results = {}
for country in ['Uganda', 'Nigeria', 'Tanzania', 'Mali', ...]:
    c = bsd.Country(country)
    x = c.food_expenditures(market='Region')
    d = c.household_characteristics(market='Region')
    y = np.log(x['Expenditure'].replace(0, np.nan)).dropna()
    r = Regression(y=y, d=d)
    results[country] = r.get_beta()
#+end_src

* Section 6: Limitations and future work (~500 words)

* Open questions for Ligon before writing starts

   microdata PPP application.  Before committing to this framing,
   we need to do a preliminary calculation on 3-4 countries to
   confirm: (a) the BSD PPPs are reasonable (agree with ICP within

   the contribution.  The risk is that a paper exists that makes

   mitigation is that even if the methodology is known, the
