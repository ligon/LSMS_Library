## Major misread (the lesson)

For PR #230 I claimed the YAML edits (`i: [ppid, hid]` etc.) were
broken-by-design — that they hit `df_data_grabber.grabber()`'s `s, f =
v` branch with `f` as a string, falling through none of the handled
types and raising `ValueError`. **This was wrong.** I traced `grabber`
in isolation without following the upstream path through `column_mapping`.

The actual path:

1. `Wave.formatting_functions` (country.py:388) auto-loads each wave's
   `_/mapping.py` and `_/<wave>.py`, registering all callables in a
   dict.
2. `Wave.column_mapping` (country.py:398) processes each `idxvars`
   entry via `map_formatting_function`, which checks
   `formatting_functions[var_name]` first. If a country function is
   registered (every one of these waves has a `mapping.py:i(value)`
   defined), the YAML's list value is wrapped as `(value,
   country_i_func)`.
3. `df_data_grabber.grabber` then takes `s, f = v` with `s = list of
   columns` and `f = country function`. Hits the `FunctionType` branch
   and runs `df[s].apply(f, axis=1)`, producing a Series of
   `format_id('-'.join(parts))`-style canonical compound strings.

So `i: [ppid, hid]` in YAML, when paired with a `mapping.py` `i()`
function, is a **first-class supported pattern** in this codebase, not
a bug. Six existing waves use it (Azerbaijan/1995, Guyana/1992,
Serbia/2007, Serbia&Montenegro/{2002,2003}, Tajikistan/1999) and
`Country(X).sample()` returns `'1-1'`-style strings as documented.

**The user's "tuples / strings-that-look-like-tuples will bite us"
warning was a design preference about the *output*, not a claim that
list-form `idxvars` are broken — the existing system honors that
preference (output is `format_id`-polished delimited strings, never
literal tuples). I conflated the two.**

What I should have done before claiming the design was broken:

- Probe one of the existing waves' `Country(X).sample()` to see what
  the system actually produces.
- Trace through `column_mapping` and `formatting_functions` before
  reading `grabber` in isolation.
- Notice that every wave with `i: [a, b]` also has a `_/mapping.py`
  that defines `i(value)` — a pattern that's not coincidence.

## #245 + #249 merge sequence (gotcha)

When merging the stack (#245 then #248), GitHub auto-closed #248 the
moment #245 merged with `--delete-branch`.  The reason: #248's base
ref was `uganda-food-acquired-phase3` (PR #245's branch), and the
delete-branch flag removed that base ref before #248 could be
auto-retargeted to `development`.  Closed-with-deleted-base PRs
can't be reopened (same stickiness as #230 earlier).

Recovery: opened **#249** as a replacement (same head branch, same
diff, base = `development`); merged it; left a pointer comment on
#248.  Net effect identical to merging #248 directly.

**Lesson**: when merging a stack with `--delete-branch`, retarget
the dependent PR's base to `development` *before* merging the
parent.  E.g.:

```
gh pr edit <child> --base development
gh pr merge <parent> --squash --delete-branch
gh pr merge <child> --squash --delete-branch
```

Or just use `--auto` flags / merge queues if available.
