* Structural differences between branches (expected, by design)

These are architectural changes introduced in development and do NOT indicate data errors:

1. *Market index*: =use_parquet= bakes the =m= (region/market) level into the stored parquet.
   =development= adds =m= at query time via =_add_market_index()= when the caller passes =market='Region'=.
   The underlying data is the same; the index structure differs.

2. *cluster_features*: =use_parquet= stored only (Region, Rural, District).
   =development= adds (Latitude, Longitude, i) from the sample frame.
   The three common columns match 100%.

3. *Shocks*: =use_parquet= has =Shock= as a DataFrame column (index=(i,t,m)).
   =development= has =Shock= as an index level (index=(i,t,Shock)).
   Values 100% match when merged on (i,t,Shock).

4. *Housing row count* (cosmetic, not a regression):
   =use_parquet= housing.py applies =.dropna()= mid-processing -> 16173 rows.
   =development= YAML path retains all households -> 24290 rows.
   For the 16173 overlapping rows, Thatched roof and Earthen floor values are 100% identical.
   This is a pre-existing difference in build strategy, not introduced by the v-migration.

5. *Nutrition build failure on development*: The =nutrition.py= script calls
   =q.groupby(['i','t','m','j'])= but =food_quantities= on development lacks the =m= index level.
   This is a pre-existing script incompatibility with the newer pipeline, not caused by the
   sample() v-migration.

6. *locality API failure on development*: =_join_v_from_sample= raises
   =KeyError: 'None of [v] are in the columns'= because =locality.py= already writes =v= as a
   data column, then =_join_v_from_sample= tries to join a second =v= from sample.  The raw
   parquet data is identical on both branches (24362 rows, v column 100% match).

* Pre-existing issues found (NOT caused by v-migration)

** nutrition.py incompatibility

=nutrition.py= line 53: =q = q.groupby(['i','t','m','j']).sum()=
=food_quantities= on development does not have =m= in its index.  Build fails with
=KeyError: 'm'=.

Background: the script was written when =food_quantities= included =m= (via the =other_features=
join).  On development, =m= is added at API time, not stored in the parquet.

Fix candidate: change groupby to =['i','t','j']= and add =m= from cluster_features.

** locality() API crash

=uga.locality()= on development raises =KeyError: "None of ['v'] are in the columns"=.
Root cause: =_join_v_from_sample= tries to add a =v= column from the sample frame, but
=locality.py= already writes =v= as a data column.  Double-addition fails.

Raw parquet data is correct and identical on both branches.

Fix candidate: in =_join_v_from_sample=, check if =v= is already a column before attempting join.

** food_prices / food_quantities: 63 duplicate index entries

Pre-existing; not introduced by v-migration.

** housing [SUBSTANTIVE IMPROVEMENT, NOT A REGRESSION]

*Note added 2026-04-10 after agent report:* the original agent
finding here was based on stale parquets in =/local/=.  A fresh
rebuild after the full Phase 3 edits landed shows a richer schema
on development than use_parquet had.

| Branch       | shape         | index           | columns                          | dtype             |
|--------------+---------------+-----------------+----------------------------------+-------------------|
| use_parquet  | (16173, 2)    | (i, t, m)       | [Earthen floor, Thatched roof]   | float64 (0/1/NaN) |
| development  | (24290, 2)    | (i, t, v)       | [Roof, Floor]                    | string[python]    |

On development, the =Roof= and =Floor= columns carry /categorical
values/ (e.g., ~Thatch~, ~Iron Sheets~, ~Cement~, ~Earth~,
~Concrete~, ~Tiles~) rather than binary indicators for a single
material.  This is a substantive improvement: downstream consumers
can dichotomize to a binary (e.g. ~(Roof == 'Thatch')~) but they
can no longer go from use_parquet's output back to the underlying
material.

The 8117 extra rows are /not/ dropped by any =.dropna()= in the old
script --- they are households lost in the old pipeline's column
construction that are now preserved by the YAML extraction path.

This honours the project design principle "preserve the detail".
Consumers of use_parquet who relied on the binary indicators can
reproduce them with a trivial one-liner; consumers who want the
material categories can now get them without touching the library.

Not a regression.  Filed as a note in CHANGELOG, not as an issue.

(The original stale-parquet finding was documented in commit
~1b0abbb0~ and corrected here in ~a2a4a531~-adjacent.)

* Conclusion

The sample() v-migration (Phases 2/3/4) did *not* break observable outputs for downstream
consumers.  Every table that can be compared on a per-row basis shows 100% value agreement.

Structural differences (index level names, presence/absence of =m=, column renames) are all
expected architectural changes documented in CLAUDE.md.

Three pre-existing issues identified during the comparison, all handled:

- =nutrition.py= KeyError on 'm': filed as GH #141.  Pre-existing
  (consequence of the earlier =other_features= removal, not the
  sample v-migration).
- =locality()= API crash from =_join_v_from_sample= conflicting
  with a legacy script that wrote =v= as a column: fixed in
  ~a2a4a531~.
- =housing= schema difference: initially flagged as "use_parquet
  dropna vs dev keep-all", but on re-verification the correct
  framing is that development produces a richer categorical
  Roof/Floor schema (values like ~Thatch~, ~Iron Sheets~, etc.)
  compared to use_parquet's binary indicators.  This is a
  substantive improvement in line with the "preserve the detail"
  principle and is not a regression.  See housing section.

The original stale-parquet observation for housing illustrates an
important lesson: when running a regression comparison against
cached outputs, always clear the cache on both sides before
building or rebuild on the side under test before comparing.
Stale parquets can produce misleading "differences" that look
like old behavior vs new behavior when they are actually
old-code-cached vs new-code-cached.

The migration can be considered safe to release from a data-correctness standpoint.
