** Cache behavior empirical verification

The planning conversation surfaced a mismatch between what
=CLAUDE.md= claimed about caching ("Subsequent calls read cache
(<1 sec). Caches auto-invalidate on source/config changes
(hash-based).") and what the code actually does.

Ligon ran a two-call =ipython= test on =niger.household_roster()=
and reported: first call slow, second fast, but *exiting ipython
and re-running* gives the same slow-first-call pattern.

Claude then ran a minimal controlled probe script
(=/tmp/niger_probe.py= — not committed) twice in fresh Python
subprocesses.  Results:

| Phase                                 | RUN 1 | RUN 2 |
|---------------------------------------+-------+-------|
| Python import =lsms_library=          | 29.5s | 36.3s |
| =Country('Niger')=                    | 0.03s | 0.03s |
| First =household_roster()=            | 21.0s | 21.9s |
| Second =household_roster()= same proc | 7.5s  | 8.4s  |

Branch counts from monkey-patched =get_dataframe=: ALL reads via
=local_file()=, zero =DVCFS= calls (the mirror has all =.dta=
files on disk from a historical =dvc pull=).  First call: 11 reads.
Second call: 4 reads (only the four wave-level source files;
the 7 "missing" calls are ancillary lookups cached on the Country
instance via =_sample_v_cache=, =_updated_ids_cache=, etc.)

Also verified: =.dvc/cache/= does *not* exist on disk.  The
initial hypothesis that DVC's on-disk content-addressed cache
was doing the within-process speedup was wrong.  The speedup
comes from (a) Country-instance ancillary caches and (b) OS page
cache for the source =.dta= files.

The plan document has full details in the "Object Categories"
section.

* Lessons learned

** Default claims in CLAUDE.md need empirical verification

The "hash-based auto-invalidation" claim in =CLAUDE.md= had
persisted through multiple sessions and multiple handoffs
without anyone checking it against what =country.py= actually
does.  It was mentally accurate for the 7 countries with
=dvc.yaml= (which is where Ligon's own interactive work happens)
and got generalized incorrectly to the whole library.  The lesson
is the same one from 2026-04-10's whole-node-allocation check:
*probe before codifying*.

** The in-tree fallback at =country.py:1372= is a silent-failure vector

It's the reason stale pre-sample-migration parquets for Uganda
and Senegal =household_roster= are currently being served to
callers on this mirror.  Removing it is a 3-line delete in the
v0.7.0 session and closes a correctness bug.

** A "cache" that writes without reading isn't a cache

The =data_root()= parquet write from the =if not stage_infos:=
branch looks like caching but isn't.  The shim that moved writes
out of the repo tree (commit unknown, probably in the v0.5.x
range) fixed the write side of the separation without fixing
the read side.  Future caching work needs both.

** DVC's on-disk content cache is not automatic

=DVCFileSystem.open()= does NOT populate =.dvc/cache/= as a side
effect of reads.  You have to =dvc pull= (or a subagent must do
it) for the persistent on-disk cache to exist.  The "fresh pip
install" scenario has no such cache and will stream from S3 on
every read until v0.7.0's cache read path lands.
