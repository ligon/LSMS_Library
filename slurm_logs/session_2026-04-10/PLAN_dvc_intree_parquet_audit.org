* Surprises

- *All 6 plain in-tree parquets are stale vs dvc.lock*.  The size mismatches
  are large: Uganda =household_roster= is 1.07 MB in-tree vs 285 KB expected by
  dvc.lock — a 3.7x bloat, consistent with the pre-sample-migration index
  including extra levels (=v=, wave-level =_/ = artifacts folded in, etc.).

- *country.py still has an in-tree fallback at line 1372*.  The CLAUDE.md says
  "writes go to data_root, not the repo tree," but reading still falls back to
  =self.file_path / "var"=.  For Uganda and Senegal, the stale in-tree parquets
  are the ONLY available cache for =household_roster= and =cluster_features= —
  the pipeline would silently serve stale data to any caller that lacks a
  data_root build.

- *lsms_ag.parquet MD5 matches its .dvc pointer exactly*.  Unlike the other 6
  files, this one is the genuine DVC-tracked artifact (pulled from S3).  It is
  not a stale local build; it just has zero Python references in the codebase.

- *Senegal has no data_root directory at all*.  =~/.local/share/lsms_library/Senegal/=
  does not exist on this machine.  Any call to =Country('Senegal').household_roster()=
  will use the stale in-tree file (or rebuild from scratch if source files are
  available).

- *The Uganda =var/README= says "one should not add files in this to git or dvc"*,
  yet the =Uganda/dvc.yaml= and =dvc.lock= explicitly list =var/*.parquet= as DVC
  stage outputs.  The README is outdated advice that predates the DVC pipeline
  setup; it conflicts with the current workflow and should be updated or removed.
