* Cache invalidation note for next session

After Phase 3 commits, anyone pulling =development= and running
=Country(X).food_acquired()= against an existing parquet cache will
see *legacy-shape* output -- the cache does not auto-invalidate when
wave scripts change (per CLAUDE.md "## Cache Behavior").  Before
running tests on a freshly-pulled tip:

#+begin_src sh
for c in Nigeria Tanzania Malawi Ethiopia Uganda; do
    lsms-library cache clear --country "$c"
done
#+end_src

or set =LSMS_NO_CACHE=1= in the session.  This caught me twice today
during Phase 4 verification -- /the cached parquets were stale Phase
3 output/ even after the Phase 3 commits had been on
=origin/development= for hours.

There is a related secondary issue: dispatched worker agents'
verification builds wrote parquets to their /worktree directory/
rather than the shared cache (because =_resolve_data_path()= can't
infer country/wave when the script lives under
=.claude/worktrees/agent-XXX/lsms_library/...=).  Net effect: the
shared cache remained legacy until the coordinator force-rebuilt
each country.  Worth a future fix in =_resolve_data_path= to handle
worktree paths correctly, but it's a low-priority developer-tooling
ergonomic, not a correctness issue.

* Harness gotchas observed today (worth handoff awareness)

** Worktree base drift on multi-agent dispatch
   Worker agents with =isolation: "worktree"= can land on the wrong
   parent commit -- empirically the harness defaults to =master= when
   the user is on =development=.  Lessons baked into
   =sucoder-skills/scrum-master-hpc/SKILL.md=:

   1. Always =git worktree list= after a multi-agent dispatch and
      verify each entry's SHA individually (commit =0b4f2a9=).
   2. Prepend an unconditional =git fetch origin <branch> && git
      reset --hard origin/<branch>= block to every worker prompt to
      self-correct (commit =9743c96=).  Avoids the
      bounce-and-redispatch round trip when the harness mis-defaults.

** Edit/Write tool path-resolution glitch
   Sub-agents reported (intermittently) that the harness's =Edit= /
   =Write= tools would route file modifications to the *main checkout*
   instead of the agent's worktree, despite =cwd= being the worktree.
   Detection: =git -C <worktree> status= shows the file clean even
   though the agent just edited it; =git -C <main> status= shows the
   diff.  Recovery: bash heredoc + =cp= to write the file directly,
   then =md5sum= to confirm both paths.

   Two of the seven worker agents I ran today hit this; one
   (Malawi 2013-14) caught and recovered cleanly, one (Malawi 2016-17)
   recovered after several Edit attempts failed.  Five others edited
   in-worktree without issue.  No clear pattern on triggers.  Worth
   reporting to harness vendor if it persists.

** Sub-agent narrative termination
   Two agents (Malawi 2004-05, Ethiopia) terminated mid-narrative
   ("Probes still running.  Let me wait for the Monitor's notification.")
   without committing despite making complete and correct edits.
   Worktree had complete diff; coordinator committed on their behalf.
   Likely a harness-side timeout or thinking-budget exhaustion.
