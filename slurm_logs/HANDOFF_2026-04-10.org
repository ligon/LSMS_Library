** Makefile parallelism (=1829595b= then corrected in =cfba5ae5=)

First commit (=1829595b=): introduced =PYTEST_ARGS= variable
defaulting to =-n auto --dist=loadfile= so =make test= and
=make release= run pytest in parallel.

Correction (=cfba5ae5=): =-n auto= turns out to be wrong inside a
cgroup-limited Slurm job because xdist's =auto= calls
=os.cpu_count()= which reports the *physical* node CPU count
ignoring cgroup affinity.  On a 56-core node where Slurm gave us
8 logical CPUs, xdist spawned 56 workers oversubscribing 8 cores
by 7×.  Fixed by introducing =PYTEST_WORKERS= which prefers
=$SLURM_CPUS_ON_NODE= and falls back to =nproc=.

* Lessons learned

** Don't enshrine inherited folklore in a skill without an empirical test

The whole-node-allocation guidance I wrote earlier in the session
was based on a verbal claim that turned out not to match reality
on =savio4_htc=.  A 10-second probe job at ~0.02 SU cost
demonstrated the actual behavior and led to the corrected skill
text.  The cost of being wrong was paid by future scrum masters
who would have followed the bad advice.  Always probe before
codifying.

** =pytest -n auto= is wrong inside a cgroup-limited Slurm job

xdist's =auto= calls =os.cpu_count()= which ignores cgroup
affinity and reports the physical node count.  Use
=$SLURM_CPUS_ON_NODE= explicitly.  The Makefile now does this via
=PYTEST_WORKERS=.

** Compute nodes have no outbound internet on Savio

=poetry build= via =make release= failed on =n0028.savio4= because
it tried to reach pypi.org for keyring/SecretStorage backend
fetches.  Worked fine after rerun on the login node.  Anything
that needs pypi (or any external HTTP) cannot be a Slurm job step.

** =poetry-dynamic-versioning= must be installed at the user level

Declaring it under =[tool.poetry.requires-plugins]= in pyproject.toml
is *not* enough in Poetry 2.x.  =poetry self show plugins= must
list it.  If not, =poetry version= reports the static =0.0.0= and
=poetry build= produces a mis-versioned wheel.  Workaround on this
machine: =pip install --user --ignore-installed
"poetry-dynamic-versioning[plugin]"=.  The =--ignore-installed= is
needed because the system Python 3.11 has read-only =pycparser=
and pip's upgrade-via-uninstall path fails on read-only files.

** DVC lock contention is real

Killed =dvc status= calls leave several stale lock files behind:
=tmp/lock=, =tmp/rwlock=, =tmp/rwlock.lock=, plus per-file
hash-named =*.lock= files.  All need to be removed before any
subsequent =dvc add= or =dvc push= will work.  =find
lsms_library/countries/.dvc/tmp/ -name "*lock*" -delete= is the
nuclear option after confirming no real dvc processes are
running.

** =dvc add= refuses files that are git-tracked

Sequence must be: =git rm --cached <files>=, then =dvc add
<files>=.  =dvc add= alone fails atomically on the first
git-tracked file and adds nothing.

** Fix directory whitespace before =dvc add=

=.dvc= pointer files store the path of the tracked file.  If you
=dvc add= a file in =Documentation /= (with space) and later
=git mv= the parent to =Documentation= (clean), the =.dvc=
pointers reference the old path and are orphaned.  Always rename
first, =dvc add= second.

** Keep literal input filenames

It's project policy to keep the original (sometimes badly
formatted) filenames of input data files such as
=Panama/_/central_american_fct - FCT.csv=.  This allows
verification against the upstream source by name without comparing
file contents.  Don't "clean up" odd filenames in the
=lsms_library/countries/*/_/= directories without a good reason.

** Architectural observation

The Uganda =locality()= feature uses column name =v= for the
parish/village name, but the =v= in =cluster_features= and
elsewhere is the EA cluster code.  Same name, two different
concepts.  Currently functional because =_join_v_from_sample=
catches the conflict, but the latent confusion is a source of
bugs (the post-v-migration =locality()= crash that
=a2a4a531= fixed was caused by exactly this).  Renaming the
locality column from =v= to =parish= or =locality_name= would
clean this up; it's a breaking change that needs a release-note
callout.
