* Conftest improvement

=ea19450b= adds =pytest --no-purge= to skip the default-mode
Uganda L1+L2+in-tree cache auto-purge.  Useful during iterative
diagnosis loops where the failures are not Uganda-extraction-
related (saves ~5–15 min per run).  Drop the flag for the final
pre-merge / pre-release verification to restore the staleness
robustness check.

Caveat: =--no-purge= alone is not sufficient to dodge the DVC
lock contention because the contention comes from =make -j12=
inside one pytest process, not from xdist's parallel workers.
The framework's =make= invocation literally hard-codes =-j12=,
overriding =MAKEFLAGS=.

* Open work

** [v0.7.3] DVC lock contention in =_ensure_dvc_pulled= [#A]

PR #212 (=6cb332a9=) migrated 115 wave scripts to call
=get_dataframe()=, which goes through =_ensure_dvc_pulled=
(=lsms_library/local_tools.py=, ~line 226).  Each wave script
acquires a =flufl.lock= around DVC's local cache.  When the
framework calls =make -j12= to build a country-level parquet
(e.g. =income.parquet=, which depends on multiple waves'
=earnings.py= + =enterprise_income.py=), all 12 sub-jobs spawn at
once and race for the lock.  Most time out with
=flufl.lock._lockfile.TimeOutError=.

Single-wave isolation (=python lsms_library/countries/Uganda/
2013-14/_/earnings.py=) builds clean, confirming the wave-script
code is correct.  The bug is purely in the parallel build
orchestration.

Fix candidates:
- Wrap =_ensure_dvc_pulled= with retry + exponential backoff
  bounded by a process-wide ceiling.
- Or serialise DVC pulls under a OS-level lock that allows
  concurrent /reads/ once the data is pulled (the second-wave-
  needs-the-same-file case is fast).
- Or change =run_make_target= to call =make -j1= when the dvc
  cache is observed to be empty for the requested files, then
  flip to =-j12= once warm.

Once landed, =income= xfail in =test_uganda_api_vs_replication.py=
will start XPASSing — drop the xfail at that point.

** [release] v0.7.2 wheel build elsewhere

The user will cut the v0.7.2 wheel from a machine with PyPI
creds.  See =.claude/skills/release/SKILL.md= (just refreshed) for
the full sequence — three things to remember on a Linux host:

1. Verify =poetry self show plugins= lists
   =poetry-dynamic-versioning=; if not, install per the skill.
2. Either install poetry into the project venv (preferred) or
   upgrade user-site =packaging>=24.2= to dodge the
   =No module named 'packaging.licenses'= failure on the system
   poetry.
3. =PYTHON_KEYRING_BACKEND=keyring.backends.null.Keyring= to
   prevent the headless-Linux keyring hang.

Tag the release commit with =git tag -a v0.7.2 -m "Release 0.7.2"=
before building so =poetry-dynamic-versioning= stamps the wheel
correctly.
