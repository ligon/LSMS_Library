* Target contract (copy the Uganda playbook)
The machinery already exists; GhanaLSS just has to /emit/ canonical wave
parquets and let the framework do the rest.

** Wave-level output (what each =<wave>/_/food_acquired.py= must write)
- Index: =(t, i, j, u, s, visit)=.  =visit= is KEPT as a country-local extra
  index level (Decision D1) — the library harmonizes the /interface/, not the
  /data/, so we do not destroy GhanaLSS's repeated-visit recall structure.
  NO =v= — the framework joins it from =sample()= at API time
  (=_join_v_from_sample=; see CLAUDE.md "sample() and Cluster Identity").
- Columns: =[Quantity, Expenditure, Price]=.
- =s= ∈ ={purchased, produced}= (GhanaLSS has no in-kind module; cf. Uganda
  which also carries =inkind=).
- =i= = household id, =j= = harmonized food item, =u= = unit.  This is the
  Uganda =food_acquired_to_canonical= contract (uganda.py:243) PLUS the local
  =visit= level.
- The canonical CROSS-COUNTRY index remains =(t,v,i,j,u,s)=; =visit= is a
  GhanaLSS-only addition.  =Country('GhanaLSS').food_acquired()= returns the
  visit detail; the composable interface is the DERIVED tables, which sum it
  out (see D1).  =Feature('food_acquired')= cross-country drops the extra
  =visit= level via =_harmonize_country_frame= (feature.py:74) — VERIFY in
  Phase 1 that it sums rather than =first()=-collapses (else re-introduces the
  data-loss this design eliminates).

** Country level

- RETIRE =food_prices_quantities_and_expenditures.py=.  =food_expenditures=,
  =food_prices=, =food_quantities= are then produced at runtime by
  =_FOOD_DERIVED= (country.py:2766) — exactly as Uganda now does post-#245.

  - =food_expenditures_from_acquired= (l.933): needs an =Expenditure= column;

  - =food_quantities_from_acquired= (l.967): needs =Quantity= + a =u= index
    level; kg-converts where a factor exists, else carries native =u=.
  - =food_prices_from_acquired= (l.1073): derives Price.
- =data_info.yml= =Index Info > index_info= for =food_acquired= → set canonical
  =(t, v, i, j, u, s)=.  (Aligns with the now-declarative no-v-join policy:
  with =v= in the canonical index the framework joins it automatically.)

* DECISIONS (resolved by maintainer 2026-06-15)
** D1. Visit / recall handling  [RESOLVED: keep visit; aggregate in derived]
Philosophy — harmonize the interface, not the data — so do NOT sum =visit=
away in =food_acquired=.  KEEP =visit= as a country-local index level so
=Country('GhanaLSS').food_acquired()= preserves the repeated-visit recall
structure.  The DERIVED tables (=food_expenditures=, =food_prices=,
=food_quantities=) aggregate over =visit=.
VERIFIED this composes with the existing machinery at no framework cost:
- =food_expenditures_from_acquired= groups by =[t,i,j,s]= (transformations.py:960)
- =food_quantities_from_acquired= groups by =[t,i,j,u,s]= (l.1036/1068)
- =visit= is in NONE of these group-bys → it is summed out automatically.
- =food_prices= derives value/qty AFTER aggregation → total expenditure /
  total quantity = the wave's average unit value (correct; price is intensive).
Bonus: declaring =visit= in GhanaLSS's index means the data IS unique at
=(t,i,j,u,s,visit)= grain → the =groupby().first()= duplicate-collapse
data-loss warning disappears (it fired only because the bespoke shape was
force-collapsed to =(t,i,j)=).  [Phase-1 check: confirm the canonical-shape
guard reads the declared index incl. =visit=.]

** D2 + D4. Missing quantities & units  [RESOLVED: price-impute in native units]
Maintainer: "our standard cleverness should fill in quantities if we have
prices in some known units — this also obviates (4)."
The "standard cleverness" is =median_price_valuation= (transformations.py:2032):
the WB/Deaton hierarchical median-price ladder — observe unit prices, take the
median in the finest geography cell clearing a ≥threshold count (=v= →
District → Region → national), impute that price to rows lacking one.  It
currently runs =value = imputed_price × quantity=; here we use the INVERSE,
=quantity = Expenditure / imputed_price=, reusing the same imputed-price ladder.
- PRIMARY price source — the dedicated COMMUNITY/MARKET price survey that
  EVERY wave fielded (=PRICE.DAT= in 1987-88/1988-89, =Data/Prices/G3PRICE.DTA=
  in 1991-92, =g7price.dta= in 2016-17, price questionnaires for 1998-99/05-06).
  This is exactly the original author's documented intent
  (=GhanaLSS/_/CONTENTS.org:137=: "a separate price survey was conducted, and
  could presumably be used to obtain quantities") and is the textbook Deaton
  unit-value input — market price per =(j, geo, u)=.  =Quantity =
  Expenditure / community_price=, with a geography fallback ladder
  (=v= → District → Region → national) for empty cells.
- SECONDARY price source — survey-reported prices already in =food_acquired=:
  the PRODUCED side (Price + qty + unit, all waves except 87-88/88-89) and
  2016-17 PURCHASES.  Used where the community survey lacks a =(j, geo)= cell.
- "Obviates (4)": impute in the NATIVE units the price survey reports, so
  quantities come back in known native units — NO bespoke
  =conversion_to_kgs.json= is needed to FILL them.  (kg conversion stays a
  SEPARATE downstream concern for =food_quantities(units='kgs')=, unchanged;
  native-=u= rows pass through as today.)
- PREREQUISITE: GhanaLSS has NO =community_prices= feature wired today (a GH
  #109 gap).  Extracting the price survey into a usable =(t, v/geo, j, u) →
  Price= table is a prerequisite for (or first step of) the imputation, and is
  a worthwhile feature in its own right.
- Items with NO price anywhere (not in the price survey, no production, not in
  2016-17 purchases) cannot be imputed → =Quantity= stays NaN, =Expenditure=
  carried.  Documented, not a bug.
*** New sub-decisions this raises (my proposals — flag if you disagree)
- *Where the imputation lives*: PROPOSE the DERIVED layer (=food_quantities=),
  NOT the wave script — keep =food_acquired.parquet= the honest survey record
  (Expenditure with NaN Quantity where unobserved), impute on read.  Cost:
  small new framework wiring (the derived path does not call
  =median_price_valuation= today).  Alternative (bake imputed Quantity into
  =food_acquired=) violates "don't mutate the data" and hides provenance.
- *Provenance*: PROPOSE an explicit marker (e.g. =Quantity_imputed= bool, or
  =s='purchased_imputed'=) so imputed quantities are distinguishable from
  survey-reported ones.  Honesty + reproducibility.
- *Native-unit basis*: imputed purchase Quantity inherits the unit of the
  observed price for that =j= (produced-side / 2016-17 modal =u=).  Assumes
  purchases of =j= are in the same unit as observed sales/production of =j=;
  documented assumption.

** D3. 1987-88 & 1988-89 (value-only)  [RESOLVED: include now; own price survey]
Include all 7 waves.  The expenditure module gives Expenditure only (no qty),
BUT both waves fielded their OWN =PRICE.DAT= community price survey — so they
are self-sufficient for imputation under D2; NO cross-wave price borrow is
needed.  Caveat (=CONTENTS.org:141=): the 1980s waves use a single-visit
"since my last visit" recall (not the ~dozen visits of 1991-92+), so their
=visit= level is degenerate (one value) — harmless, but note it for
cross-wave comparability.

** D5. Household =i=-key (fixes the NaN-=v= bug)  [PROPOSAL: route through the wave i() helper]
Confirmed root cause of "100% NaN v" (GH #256 pattern) — =food_acquired.py=
builds =i= differently from =sample=/=roster=:
- 1991-92: =format_id(clust)+format_id(nh)= → ="11"= vs sample/roster
  ="101"= (no zero-pad on nh).
- 1998-99: =f"{clust:d}/{nh:02d}"= → ="1/01"= vs sample/roster ="101"=
  (spurious "/" separator).
- 2005-06: reconstructs =hhid= from clust/nh vs sample/roster reading the
  /pre-composed/ source =hhid=.
PROPOSAL: delete the bespoke =i= construction in every =food_acquired.py= and
call the wave's canonical =i()= (mapping.py / ghanalss.py) — the same helper
=sample= and =household_roster= use.  This makes the keys identical by
construction and removes the per-script drift permanently.  Re-verify with a
=set(sample.i) ∩ set(food.i)= intersection check per wave.

** D6. Food-item harmonization  [PROPOSAL: map both schemes to harmonize_food]
Purchases use =Code_9b= (=Code_12A= in 87-88/88-89), production uses =Code_8h=
(=Code_12B=).  Both must map to the canonical =j= via the country's
=harmonize_food= / =food_items= table.  Gap: =food_items.org= harmonization is
itself a GH #109 TODO.  PROPOSAL: scope the =Code_9b/8h → Preferred Label=
mapping audit into Phase 1 (it is a prerequisite, not optional); the
=Code_12A/12B= mapping rides on the D3 decision.

* Tooling context (v0.8.0, merged to development 2026-06-15 — PR #454)
Two just-landed changes materially help this migration:
- *Automatic content-hash cache invalidation*: the L2 cache hash now covers
  =data_info.yml=, the wave/country modules (=ghanalss.py=, =mapping.py=), the
  =_/{table}.py= scripts, =data_scheme.yml=, the =Makefile=, and =.org= build
  inputs (=food_items.org=, =categorical_mapping.org=) + the DVC sidecar md5 of
  sources.  So editing any of these during the wave rewrites AUTO-invalidates
  and rebuilds — no manual =cache clear= per iteration (kills the old
  stale-L2-wave-parquet-shadows-a-script-fix gotcha).  Caveat to respect:
  script-path L2-wave parquets are written hashless and self-invalidate only
  via the L2-country eviction path — if the country table is never
  materialized (partial cache), a hashless wave parquet can still serve stale
  (residual F1 in =dvc_object_management.org=); during the loop, build the
  country table (or =LSMS_NO_CACHE=1=) when verifying a wave-script edit.
- *GhanaLSS Makefile fixed* (=cd1c95cd=/=86a10d10=): generic rule now depends on
  =ghanalss.py= (was a stale =ghana.py=) and the =household_roster=
  prerequisite is corrected — the Make backend our Phase-2 relies on is now
  sound.

* Risk register

  a wrong visit/source reshape could keep dropping them invisibly.  Mitigation:
  assert row-count conservation across the reshape; no =groupby().first()=.
- MED — food-code harmonization gaps (unmapped codes → '' =j= → dropped).
  Mitigation: leak audit per wave (cf. the =food-acquired/units= skill).
- MED — id_walk panel chains across 7 waves (1987→2017) may not be defined;
  cross-wave =i= continuity is a separate question (verify, don't assume).
- LOW — kg conversion deferred (D4) → =food_quantities(units='kgs')= returns
  native-=u= rows until the FCT lands; acceptable and documented.
